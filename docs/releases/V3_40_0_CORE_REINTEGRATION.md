# PitGuard V3.40.0 核心回归与系统整合

## 背景

V3.39 日志显示核心接口已经可用，但页面仍会产生重复的渐进式配置写入、同步诊断和高级审查请求。典型问题包括：渐进式配置连续返回 409、同步计算修复耗时 12.6 秒、工业成熟度/问题中心/高级套件单次读取耗时 5–7 秒，以及计算追踪将结构化检查值强制转换为浮点后返回 500。

## 本次重构

1. 默认前端改为五阶段核心工作台；
2. 新增 `core_design` 后台任务，串联围护方案、当前方案计算和配筋；
3. 新增轻量 `design/core-status` 接口；
4. 修复结构化规范检查值导致的计算追踪 500；
5. 渐进式配置冲突改为服务端自动重基，不再循环 409；
6. 候选默认上限为三个，完整计算默认仅当前方案；
7. 高级资格、成熟度、监测和基准模块移出主流程；
8. 138 份迭代文档归档，主文档收敛为当前产品的五份核心说明。

## 核心数据目标

- 网页工作区保持有界；
- 当前完整计算结果默认保留 1 个；
- 候选方案默认不超过 3 个；
- 重型地质、阶段结果和钢筋几何全部外部化；
- 页面打开不触发完整项目重组和工程计算。

## 默认运行模式

启动脚本默认设置 `PITGUARD_PRODUCT_MODE=core`，只注册核心路由。需要迁移历史扩展功能时可显式设置 `PITGUARD_PRODUCT_MODE=full`，但扩展模式不属于核心验收链路。

## 核心链路实测

`scripts/smoke-core-workflow.py` 已端到端完成项目创建、钻孔导入、轮廓录入、三方案生成、当前方案施工阶段计算和均衡配筋。测试耗时 4.324 s，生成 1 个完整计算结果和 240 项配筋检查；项目快照 1.82 MB、工作区 1.51 MB、外部成果 0.67 MB，总逻辑数据约 2.49 MB。

详细功能和文档审计见 `V3_40_0_FUNCTION_DOCUMENT_AUDIT.md`。
