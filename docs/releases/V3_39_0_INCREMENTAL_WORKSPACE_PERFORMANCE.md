# PitGuard V3.39.0 增量式工作区与后端性能恢复

## 1. 问题定位

V3.38 的大项目工作区已经避免把完整快照直接送入 API，但围护结构页面仍会在挂载时并发请求设计资格、渐进式配置、平面识别、布设审计、构件深化、计算资源估算、存储健康和外部对象清单。多个接口各自执行以下过程：

1. 从 SQLite 读取并反序列化 `workspace_data`；
2. 运行 Pydantic 项目模型校验；
3. 重复执行平面分类与体系目录生成；
4. 重复执行坐标、地质覆盖和几何一致性审计；
5. 部分路径重新运行支撑拓扑全量质量检查；
6. 外部对象清单路径重新解析完整 `projects.data`。

在 510 MB 完整快照、17 MB 左右工作区和 168 根支撑的工程中，并发请求会竞争 CPU、SQLite 连接和 Python 堆内存。前端 5～8 秒超时随后中止请求，并对 GET 自动重试一次，使后端在原计算尚未结束时接收第二批相同计算，形成超时放大。

## 2. 新的首屏请求模型

围护结构页面现在只发起一个首屏请求：

```text
GET /api/projects/{project_id}/design/workspace-bootstrap
```

该请求一次完成：

- 工作区项目模型加载；
- 平面形状诊断；
- 支撑体系目录；
- 设计资格摘要；
- 渐进式设计会话；
- 动态资源策略；
- 项目存储健康摘要。

服务端将中间结果显式复用，避免资格和渐进式配置内部再次进行形状识别、体系生成和资源采样。旧接口继续保留，内部从同一 bootstrap 快照取值。

## 3. single-flight 与工作区模型缓存

### 3.1 项目级 single-flight

同一个项目、修订号、工作区体积和渐进式配置版本对应唯一缓存键。并发请求中只有第一个线程执行分析，其余线程等待并复用结果。默认缓存参数：

```bash
PITGUARD_DESIGN_BOOTSTRAP_CACHE_SIZE=24
PITGUARD_DESIGN_BOOTSTRAP_CACHE_TTL_SECONDS=90
```

项目修订或渐进式配置版本变化时缓存键自动变化；保存配置后主动清除旧快照。

### 3.2 已校验工作区模型缓存

`ProjectRepository.require_workspace()` 先读取仅包含修订号和工作区体积的元数据。修订未变化时直接复用已完成 JSON 解析和 Pydantic 校验的 `Project` 模型。默认配置：

```bash
PITGUARD_WORKSPACE_MODEL_CACHE_SIZE=16
PITGUARD_WORKSPACE_MODEL_CACHE_TTL_SECONDS=120
```

模型缓存只用于只读工作区接口。完整计算和写操作继续由独立 worker 或完整项目路径执行。

## 4. 资格检查分层

首屏设计资格采用 `summary` 拓扑证据：

- 已选候选的硬约束结果；
- 当前有效计算结果中的支撑质量快照；
- 候选可行数量和受控阻断状态。

首屏不再重新运行逐构件、逐交点的全量支撑质量检查。以下场景仍执行完整质量检查：

- 用户展开高级布设审计；
- 候选生成后台任务；
- 完整施工阶段计算；
- 正式发行门禁。

资格响应会标记证据来源，例如：

```text
workspace_summary
current_calculation_quality_snapshot
live_full_quality_evaluation
```

该分层减少重复计算，同时避免把摘要证据误报为新的完整验算。

## 5. 前端渐进式加载

### 首屏加载

只加载：

- 项目工作区；
- 存储健康；
- 设计 bootstrap。

### 用户展开后加载

“高级操作与人工锁定”展开后才加载：

- 布设方案设计器完整性审计；
- 支撑构件深化；
- 计算资源估算。

### 数据索引展开后加载

外部对象清单只在点击“查看数据索引”后读取。关闭状态下不访问 `/artifacts`。

## 6. 外部对象清单优化

V3.38 的 `/artifacts` 会解析完整 `projects.data`，在大型项目中可能单独占用数十秒。V3.39 改为：

1. 优先从轻量 `workspace_data.advancedEngineering.artifactStorage` 读取引用；
2. 历史工作区未保留引用时，直接扫描项目对象目录；
3. 不再为了显示索引而加载完整项目快照。

目录扫描只能恢复对象类型、哈希和存储体积；缺失的逻辑体积和业务元数据保持为空，不编造数据。

## 7. 超时与重试策略

首屏 bootstrap 超时调整为 30 秒，并关闭自动重试。常规情况下，由于共享计算和缓存，响应应明显低于该值。30 秒属于故障保护，不作为目标性能。

- 存储健康：15 秒、无自动重试；
- bootstrap：30 秒、无自动重试；
- 深化审计：60 秒、无自动重试；
- 后台任务仍通过任务轮询显示真实进度。

关闭首屏重试可以避免用户网络中止后产生第二个相同重计算。

## 8. 前后端职责

### 前端

- 展示工作区与已有结果；
- 维护渐进式设计配置；
- 按用户动作加载高级分析；
- 提交后台任务和显示进度。

### API

- 返回轻量工作区和 bootstrap；
- 维护短期分析缓存；
- 编排任务；
- 不执行完整项目重组、完整计算或重型导出。

### worker

- 完整项目重组；
- 候选优化；
- 全量拓扑质量检查；
- 施工阶段计算；
- 配筋、节点、IFC、CAD 和报告导出。

## 9. 验证

- V3.36～V3.39 后端专项回归：20 项通过；
- V3.39 新增缓存、延迟拓扑检查、工作区缓存失效和无全快照对象索引测试：4 项通过；
- 前端测试：17 个测试文件、29 项通过；
- TypeScript 编译与 Vite 生产构建通过；
- Three.js 公共包仍约 524 kB，属于后续三维查看器拆包事项。
## 10. 本地基准复测

在包含 20 点台阶形轮廓、144 根支撑、约 1.04 MB 工作区投影的测试工程上，单进程本地复测结果如下：

| 阶段 | 首次请求 | 缓存命中 |
|---|---:|---:|
| 工作区解析与模型校验 | 74.15 ms | 已复用 |
| 平面识别 | 3.02 ms | 已复用 |
| 资格摘要 | 23.63 ms | 已复用 |
| 渐进式会话 | 0.11 ms | 已复用 |
| bootstrap 总耗时 | 104.74 ms | 2.92 ms |

该结果用于验证重复计算已被消除，不代表 17 MB 工作区和不同硬件上的固定承诺。生产环境应通过 bootstrap 响应中的 `performance` 字段、API P95 和 slow-bootstrap 日志持续观测。
