# PitGuard V3.36.0：通用设计资格与结构体系决策

## 1. 迭代目标

V3.36.0 处理两个相互关联的问题：

1. 大项目完整快照超过 API 交互加载阈值后，工作台仍应能够读取几何、地质、支撑候选摘要和设计资格，不应以一条 413 错误使方案面板整体失效；
2. 当现有墙—墙轴压支撑模型无法闭合时，界面应明确进入结构体系选择，不再把诊断候选显示为空白 A/B/C 设计方案。

本轮采用通用资格框架，不为某一条长条形轮廓编写专用前端或专用门禁。平面识别仍由统一分类器负责，体系候选由可扩展目录生成，资格门禁与平面类型解耦。

## 2. 总体架构

设计流程改为六级资格链：

1. **项目数据工作集**：判断工作区投影是否可用、完整快照是否允许在 API 进程加载；
2. **围护几何一致性**：检查轮廓闭合、墙段缺失和孤立墙段；
3. **坐标与地质覆盖**：检查钻孔—基坑坐标关系、模型设计域覆盖和外推风险；
4. **支撑体系与传力拓扑**：检查是否形成满足硬约束的候选，或是否进入受控阻断；
5. **当前方案计算证据**：检查计算结果、拓扑版本和计算合同是否有效；
6. **深化与正式交付**：检查节点、配筋、专业校审和正式发行闸门。

前端根据资格链进入三种模式：

- `normal`：允许按标准流程生成候选、完整计算和深化；
- `degraded`：工作区可继续使用，但存在存储、坐标、地质或证据预警；
- `diagnostic`：只允许诊断、体系选择和修复，不把候选卡解释为可计算方案。

## 3. P0：交互与稳定性闭环

### 3.1 大项目工作区读取

以下只读接口改用 `workspace_data`：

- 平面形状诊断；
- 支撑设计审计；
- 支撑深化摘要；
- 计算资源估算；
- 新增的设计资格与体系候选接口。

完整项目仍只允许独立 worker 和明确的 `profile=full` 调用。网页工作区不再因完整快照超过 96 MB 而无法读取支撑方案摘要。

### 3.2 项目存储压缩任务

新增后台任务：

```text
storage_compaction
```

执行过程：

1. 直接读取 SQLite 持久化 JSON；
2. 外部化地质面、阶段计算、候选完整计算、逐根钢筋和重型缓存；
3. 重新生成有上限的工作区投影；
4. 写回体积、对象数量和压缩前后指标；
5. 写入压缩前后体积、对象数量和项目审计事件；
6. 前端刷新项目工作区。

该任务不在 API 进程中构造完整 `Project` 对象，避免大对象多次复制。

### 3.3 诊断候选显示规则

当候选带有：

```text
capabilityOutcome = controlled_block
```

A/B/C 面板执行以下规则：

- 诊断候选不进入设计卡列表；
- 不渲染空白平面作为“方案 A”；
- 显示“当前没有具备计算资格的 A/B/C 方案”；
- 引导用户进入体系级候选区；
- 只有满足拓扑硬约束的候选才能进入完整计算按钮；
- 计算页按钮和独立 worker 使用同一资格门禁，前端绕过或旧客户端提交也无法启动无资格计算。

## 4. P1：通用结构体系候选

### 4.1 结构体系目录

V3.36.0 建立统一目录，当前包含：

| 体系 | 自动化状态 | 计算模型边界 |
|---|---|---|
| 主轴短跨墙—墙对撑 | 自动生成 | 纯轴压墙—墙体系 |
| 短跨对撑与墙—墙角撑混合 | 自动生成 | 角撑两端独立落墙 |
| 分区墙—墙对撑与显式转接区 | 初步生成 | 转接区需环梁、分隔墙或框架 |
| 闭合内环梁与径向支撑 | 自动生成后完整复核 | 环梁具备平面内弯剪刚度 |
| 中心岛法或留土分区施工 | 需定义体系 | 显式施工阶段与换撑 |
| 环桁架或多环支撑体系 | 需定义体系 | 梁/框架模型 |
| 显式双向平面框架 | 需专项模型 | 轴力、弯矩、剪力和节点刚度 |
| 显式空间框架 | 需专项模型 | 空间自由度和局部节点 |
| 分仓开挖与分隔墙 | 需定义体系 | 分隔墙稳定和差异变形 |
| 双层环梁、径向伺服支撑 | 需专项模型 | 设备与失效工况专项设计 |

目录字段包括：

- `generationMode`；
- `modelClass`；
- `prerequisites`；
- `hardBoundaries`；
- `candidateReadiness`；
- `nextAction`。

新增体系只需扩展目录和对应生成器，不需要重写前端状态逻辑。

### 4.2 平面类型覆盖

体系候选直接消费统一形状分类结果，已验证：

- 长条形矩形；
- 长条形台阶平面；
- 普通矩形和旋转四边形；
- 近方形和一般凸多边形；
- L/T/U/C/Z/H 及一般凹多边形；
- 圆形、椭圆形和规则多边形井筒；
- 含中心岛、局部坑和障碍物的工程。

框架不以 `slender_stepped_strip` 作为唯一分支。该类型只是统一分类器中的一个算例。

### 4.3 显式体系选择

支撑优化任务新增：

```json
{
  "topologyFamily": "direct_grid | hybrid_diagonal | ring_radial | zoned_direct"
}
```

约束：

- 只允许已经接入自动几何生成和拓扑预检的体系；
- 未实现的中心岛、环桁架、双向框架等体系返回 `manual_review`；
- 禁止把未实现体系静默转换为普通直撑；
- 候选仍需经过非法穿越、端点、围檩跨、障碍和传力路径硬约束。


### 4.4 通用围檩缺失支点修复

原算法只有在超限墙面能够直接找到相对平行墙时，才会保留一个修复目标。端墙、旋转边、台阶边和一般凸角附近的超限跨可能因此只剩诊断候选。V3.36 将“需要支点的位置”和“最终采用的构件体系”解耦：

1. 从每个超限围檩区间生成去重的目标站；
2. 优先搜索相对墙直撑；
3. 无相对墙时搜索两端均落墙的端墙斜撑；
4. 再搜索具有独立墙节点的平行角撑；
5. 所有新增构件继续执行坑内包含、障碍避让、零非法交叉、墙节点净距和端点真实性检查。

目标站算法只消费墙面长度、既有支点和配置限值，不判断 `slender_stepped_strip` 等具体类型，因此可用于凸多边形、旋转平面、台阶平面和凹形分区的局部围檩跨闭合。

## 5. P1：坐标与地质一致性

新增坐标审计包括：

- 基坑包围盒；
- 钻孔包围盒；
- 两者中心偏移；
- 归一化中心偏移；
- 平面交叠率；
- 平面尺度比；
- 项目坐标系类型和原点；
- 基坑是否显式定位或已按地质居中；
- 建议平移预览。

处理原则：

- 系统不自动修改工程坐标；
- 建议平移只用于人工核对；
- m/mm 等两个数量级以上尺度异常进入硬阻断；
- 无交叠但地质模型通过外推覆盖时，计算可受控继续，正式发行保持复核；
- 正式工程仍需测量控制点或坐标转换文件。

地质覆盖审计在工作区模式下优先使用完整地质面，缺失时使用 `surfacePreviews`，避免重型曲面外部化后误报“无地质模型”。

## 6. P2：后续深化接口

V3.36.0 已建立体系目录和资格门禁，以下能力仍需继续实现：

1. 中心岛、环桁架、多环、显式双向框架的真实构件生成器；
2. 结构体系切换后的施工阶段自动编排；
3. 框架/环梁节点的半刚性和二阶效应；
4. 分区开挖、换撑和永久结构协同；
5. 体系级材料量、工期、出土通道和风险联合评价；
6. 坐标控制点导入、二维刚体配准和人工确认写回；
7. 设计资格结果写入计算书首页、方案台账和正式发行记录。

## 9. 工程边界

V3.36.0 的体系候选表示“系统具备生成或定义该受力体系的路径”。它不自动等同于：

- 完整施工阶段计算通过；
- 节点局部验算通过；
- 配筋与构造深化完成；
- 正式施工图允许发行。

所有正式结论继续由 V3.35 建立的证据分级和正式发行门禁控制。
