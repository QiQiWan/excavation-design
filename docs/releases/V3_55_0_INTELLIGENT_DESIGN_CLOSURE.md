# PitGuard V3.55.0 智能设计闭环与完整配筋合同

## 1. 本次结论

V3.55 将计算校核从“只展示结果”升级为可执行的设计反馈闭环：

1. 运行当前施工阶段计算；
2. 按墙段、冠梁、围檩和支撑定位控制验算；
3. 对可安全自动调整的构件执行局部补强；
4. 使用同一施工阶段和设计输入重新计算；
5. 数值闭合后进入配筋，不能自动处理的事项转为可定位的专业复核卡片。

自动闭环不会降低荷载、提高土体参数、修改水位或覆盖工程师锁定的施工顺序。涉及软弱下卧层、降水、换撑条件和专项节点的判断仍由工程师确认。

## 2. 三个故障的根因

### 2.1 已通过计算仍显示“结构尚未闭合”

软弱下卧层筛查指标采用：

`软弱层指标 = c / 20 + φ / 25 + γ' / 10`

该指标越大越安全。旧展示层把它误识别为“越小越安全”的风险指标，导致实际 `2.51 / 1.35` 被倒算为 `1.35 / 2.51 = 0.54`。因此原始计算已经通过、硬失败为 0，智能闭环仍误判为安全储备不足，并继续阻断配筋。

V3.55 按规范目录规则明确保存不等式方向，软弱层指标采用最小值控制。专业筛查警告与数值承载力失败也被分流：

- 数值未达到规范或项目储备：进入自动补强；
- 原始规则明确为警告、人工复核或初步筛查：保留提示，但不静默增厚构件，也不把结构重新判为失败；
- 完整验算目录中的缺资料占位项：继续在验算矩阵展示，不计入本轮计算的专业复核数量。

### 2.2 配筋应用后仍需刷新才识别闭合

旧流程先生成深化门禁，再把配筋方案写入构件。第一次响应中的支撑与梁配筋合同为空，页面可能显示“结构未闭合”；刷新后重新读取项目才变为闭合。

V3.55 在纵筋、箍筋、构造筋、拉结筋和附加筋全部写回后，立即重建深化门禁。第一次“生成并应用配筋方案”的响应就包含真实合同数量，不再依赖刷新页面。

### 2.3 水平支撑箍筋和构造筋消失

旧链路存在两处数据损失：

- 配筋列表按问题状态过滤后，正常构造钢筋未进入展示数据；
- 部分重算路径只回写纵筋和一组箍筋，覆盖了原有的完整配筋数组。

V3.55 将水平支撑配筋定义为不可缺项的五类合同：

| 类型 | 工程含义 |
|---|---|
| 纵筋 | 轴压承载力及最小配筋控制 |
| 箍筋 | 端部加密区和跨中普通区抗剪、约束 |
| 构造分布筋 | 支撑侧面裂缝分散与骨架稳定 |
| 拉结/架立筋 | 保持纵筋净距和钢筋骨架尺寸 |
| 附加筋 | 锚固、错开搭接和节点局部加强 |

计算重算、方案应用和页面预览统一读取完整五类合同；问题筛选只改变表格视图，不再改变构件正式数据。

## 3. 智能闭环策略

系统对每个物理对象展开验算证据，不再只补强汇总表中的首个控制构件。自动动作包括：

- 墙体强度或刚度不足：只增厚控制墙段并重新计算内力与配筋；
- 冠梁、围檩不足：增大控制梁宽/梁高并重新计算受弯、受剪和挠度；
- 支撑轴压或稳定不足：调用支撑深化设计器优化截面和无侧向支承条件；
- 嵌固、隆起、渗流或整体稳定：给出逐墙墙趾加深候选，允许不同墙段采用不同设计长度；
- 难以自动闭合：提供“刚度优先”和“经济分区”两种继续迭代策略，并列出需要人工确认的资料、填写页面和处理建议。

自动循环默认最多 5 轮、硬上限 8 轮。每轮保存控制对象、补强前后尺寸、原因和计算结果，便于工程师审计或撤销设计决策。

## 4. 人机交互

计算页新增“校核—补强—复算智能闭环”面板：

- 显示已执行轮数、结构硬失败、项目储备不足和专业复核分类；
- 展开每轮可查看具体补强构件、原因和尺寸变化；
- 可自动处理的建议提供“应用并继续闭环”；
- 需人工处理的建议提供“定位并处理”，直接跳转到施工阶段、地质或围护方案页面；
- 数值已经闭合但仍有专业复核时，明确显示“数值闭合，复核项已列出”，不再使用“结构尚未闭合”。

同类逐构件提醒按规则归组显示，例如 57 根支撑的换撑路径复核显示为一类问题，同时保留受影响对象数量和代表构件。

## 5. 冠梁与配筋深化闭合

冠梁不再只有几何对象。计算器从施工阶段墙顶剪力包络形成冠梁线荷载，完成受弯、受剪、挠度、设计内力和五类钢筋回写。深化入口同时检查：

- 当前计算合同与施工阶段结果完整；
- 冠梁、围檩正式设计结果存在；
- 支撑和梁五类配筋合同完整；
- 截面调整后的计算已经更新；
- 构造、锚固、接头和空间问题可进入 P3 继续处理。

构造复核属于 P3 的工作内容，不再倒置为进入 P3 之前必须完成的结构阻断。

## 6. 验证结果

## 7. 工程边界

本次闭环属于设计辅助与校核反馈，不替代注册岩土/结构工程师对规范适用性、地质参数、地下水、施工组织、节点构造及正式图纸的签审。系统不会通过修改不利输入来换取“通过”，所有自动补强均为有上限、单调增强并可追溯的设计动作。
