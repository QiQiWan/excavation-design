# V3.40.0 功能、文档与界面整合审计

## 1. 审计结论

V3.39 之前的主要问题已经从“缺少能力”转为“能力叠加过多”。连续版本分别增加资格、渐进式配置、成熟度、问题中心、监测、专家优化、绘图规则和多配置交付，但这些模块重复读取同一项目并重复解释相同工程状态，造成了三类后果：

1. 核心设计路径被辅助审查路径包围，设计人员无法快速完成输入、方案、计算、配筋和交付；
2. 同一项目在一个页面内被多个接口重复解析和检查，产生 409、5–12 秒同步请求和 500；
3. 历史计算、候选完整结果、地质网格、钢筋几何和导出中间体混入主项目快照，普通项目也可能增长到数百 MB。

V3.40 将默认产品重构为五段核心链路，其他研究性和管理性能力只在显式 `full` 模式下注册。

## 2. V3.39 基线规模

| 对象 | 数量 |
|---|---:|
| Python 文件 | 177 |
| 服务模块 | 59 |
| 路由文件 | 22 |
| API 端点 | 约 180 |
| React 组件和页面 | 44 |
| Markdown 文档 | 138 |
| 原项目工作台 | 1888 行 |
| 原全局样式表 | 2189 行 |

文件数量本身不是错误。问题在于默认运行时同时暴露并调用了过多模块。

## 3. 重复功能分组

### 3.1 工程状态重复解释

以下模块都在回答“当前工程能否继续”：

- 设计资格；
- 渐进式设计；
- 工业成熟度；
- assurance gap analysis；
- issues；
- advanced suite；
- formal gate。

整合结果：默认页面只读取 `design/core-status`。正式成果生成时继续调用必要的计算与交付门禁，成熟度和高级审查不再参与首屏。

### 3.2 围护方案重复生成

历史上并存：

- `design_service` 的直接生成；
- `support_layout_optimizer` 的候选搜索；
- `integrated_retaining_optimizer` 的墙长、墙趾和支撑联合候选；
- `expert_design` 的专家候选；
- 独立墙长和墙趾优化器。

整合结果：核心模式只保留“轮廓识别 → 最多 3 个支撑候选 → 采用 1 个方案”。联合墙长、墙趾和专家体系保留为扩展工具，不参与普通设计。

### 3.3 计算与恢复入口重复

历史上并存：

- 一键计算；
- diagnose-and-repair；
- calculation assurance；
- calculation trace；
- A/B/C 并行完整计算；
- benchmark calculation。

整合结果：核心模式默认只计算当前采用方案。诊断修复不再作为主流程同步接口自动执行；A/B/C 完整计算和基准算例退出默认页面。

### 3.4 配筋与钢筋几何重复存储

历史上配筋方案、逐根钢筋、制造数据、IFC 钢筋和图纸钢筋可能同时保存在项目快照中。

整合结果：项目只保留配筋方案、验算摘要和外部对象引用；逐根钢筋几何仅在查看或导出时生成并存入外部成果。

### 3.5 文档重复

多个版本的发布说明、P0–P3 报告和功能说明同时作为在线文档存在，用户无法判断哪一份有效。

整合结果：138 份 Markdown 文档归档为一个 ZIP，主文档仅保留当前工作流、工程边界、运行存储、测试限制、功能审计和本次发布说明。

## 4. 当前默认产品边界

### 核心 API 路由

默认 `PITGUARD_PRODUCT_MODE=core` 只注册：

- 认证；
- 项目；
- 后台任务；
- 钻孔；
- 地质；
- 基坑轮廓；
- 围护设计；
- 计算；
- 配筋；
- 导出。

实测 API 路由对象由 full 模式 180 个降至 core 模式 124 个，唯一 API 路径由 152 个降至 100 个。更重要的变化是高级路由模块不再导入，主页面不再调用它们。

### 可选扩展

`PITGUARD_PRODUCT_MODE=full` 才注册：

- 工业成熟度；
- 高级监测和数字孪生；
- benchmark；
- 专家墙长和墙趾优化；
- 绘图规则优化；
- 问题中心；
- 多配置成果管理。

这些模块保留代码兼容性，不构成 V3.40 核心链路的验收范围。

## 5. 前端精简结果

默认工作台只显示：

1. 工程输入；
2. 围护方案；
3. 计算验算；
4. 配筋深化；
5. 成果交付。

默认界面已移除：

- 系统可靠性长条；
- 数据工作区说明；
- 规范过程矩阵；
- 八阶段渐进式配置；
- P0–P3 成熟度；
- 问题中心；
- 高级监测；
- 多组重复资格卡；
- 默认 A/B/C 三套完整计算；
- 多配置 IFC 和制造数据入口。

生产构建由 50 个转换模块降为 30 个。核心构建不再包含 208 kB 的旧工作台、524 kB 的 Three.js 包和高级查看器；这些源码仍留作历史兼容，但不进入默认生产包的浏览器依赖图。

## 6. 核心数据约束

- 候选方案最多 3 个；
- 当前完整计算默认 1 个；
- 历史完整计算只保留摘要或外部引用；
- IDW 曲面、施工阶段明细、逐根钢筋和导出文件外部化；
- 网页读取工作区投影；
- API 不因打开页面而重组完整项目；
- 重型计算和导出只由后台任务执行。

## 7. 实际最短链路验证

使用 4 个钻孔、3 类地层、40 m × 20 m 闭合轮廓运行：

| 项目 | 结果 |
|---|---:|
| 核心任务状态 | success |
| 总耗时 | 4.324 s |
| 候选方案 | 3 |
| 当前完整计算结果 | 1 |
| 支撑 / 立柱 | 36 / 4 |
| 配筋检查 | 240 |
| Fail / Warning | 0 / 5 |
| 完整项目快照 | 1.82 MB |
| 网页工作区 | 1.51 MB |
| 外部成果 | 0.67 MB |
| 总逻辑数据 | 2.49 MB |
| 外部对象 | 4 |

该结果用于证明五段核心链路能够在同一版本中完整运行。耗时与工程规模及硬件相关，不作为大型工程性能承诺。

## 8. 被判定为过期或冲突的历史要求

旧测试曾要求 L 形平面的一键轴压墙—墙支撑不得返回 fail。后续版本又明确要求：没有环梁、分隔墙、中心岛或显式框架转接时，L/T/U/C 等深凹平面应受控阻断。两项要求互相冲突。

V3.40 采用后者：系统可以识别并形成方案草案，但不会用无定义的 T/Y 节点强行通过完整计算。该策略属于工程边界，不再以“自动生成任意支撑即可通过”作为成功标准。

## 9. 当前限制

- 深凹异形基坑缺少明确转接体系时仍需要工程师选择环梁、分隔墙、中心岛或框架方案；
- 正式 CAD 输出依赖 `ezdxf`；
- 扩展模式仍包含大量历史模块，其正确性不等同于核心链路通过；
- 旧工作台源码保留用于迁移参考，默认构建和默认页面不加载。
