# PitGuard V3.46.0：P0–P2 工程闭环实施报告

## 1. 版本目标

V3.46.0 在 V3.45.0 六阶段核心流程上继续推进 P0–P2，目标不是扩充外围功能，而是补齐以下工程链路：

- P0：旧项目升级、设计依据确认、验证证据状态、故障定位和真实工程迁移；
- P1：工程等级模板、作用分组、多方案 Pareto 比选、型钢目录和配筋可施工性；
- P2：开裂刚度、节点半刚性、刚域、初始缺陷、长期效应、不利工况筛查和局部节点子模型。

版本仍采用“API 轻量编排 + 一次性隔离 worker + 外部重型成果”的运行架构。所有新增计算结果均标注计算方法、证据状态和适用边界，不用代理值填充未实现的正式验算。

## 2. P0：工程可用性收口

### 2.1 旧项目透明迁移

新增设计基准迁移服务和脚本：

```bash
python scripts/migrate-v346-design-basis.py --dry-run
python scripts/migrate-v346-design-basis.py
```

迁移行为：

- 补充缺失的设计基准模板编号；
- 补充作用分组目录；
- 补充项目安全系数目标；
- 补充位移限值草案；
- 保留原项目的 `designBasisConfirmed` 状态；
- 未经设计人员确认时继续阻断正式计算和交付。

系统不会因为版本升级而自动确认设计基准。

### 2.2 规范校核证据状态

强度、刚度和稳定性检查统一使用以下证据状态：

- `calculated`：当前施工阶段计算已生成有效数值；
- `missing_input`：模块已实现，但缺少勘察、截面、材料或控制工况输入；
- `not_calculated`：具备输入但尚未运行当前方案计算；
- `not_implemented`：当前计算内核未覆盖，必须人工或专项软件复核；
- `manual_review`：存在工程判断、地方标准或专项构造要求。

工作区同步统计各状态数量。界面和计算书不再把“缺失数值”显示为通过。

### 2.3 项目级审计

新增项目审计脚本：

```bash
python scripts/audit-project-p0-p2.py --project-id <PROJECT_ID>
```

默认输出：

```text
runtime/diagnostics/p0-p2-project-audit.json
```

审计内容包括：

- 设计基准、阻断项和存储状态；
- 规范检查证据覆盖率；
- A/B/C 多方案和 Pareto 结果；
- 型钢截面目录和配筋可施工性；
- 当前分析模型合同；
- 不利工况筛查；
- 全局耦合体系数量和局部节点检查数量。

### 2.4 运行诊断补充

V3.46 新增以下结构化日志流：

- `design-migration.jsonl`：设计基准迁移和确认状态；
- `verification-coverage.jsonl`：规范检查证据覆盖；
- `analysis-scenarios.jsonl`：施工阶段、不利情景和控制指标；
- `rebar-detailing.jsonl`：配筋草案、拥挤、锚固、搭接和套筒检查。

日志只保存摘要、数量、状态和内存指标，不写入完整模型、计算矩阵或逐根钢筋坐标。

## 3. P1：规范参数与多方案深化

### 3.1 工程等级模板

提供三套待确认模板：

| 模板 | 适用定位 | 主要特点 |
|---|---|---|
| 一级基坑 / 高敏感环境 | 高重要性、复杂场地或敏感周边 | 较高重要性系数、稳定储备和更保守的开裂刚度 |
| 二级基坑常规场地 | 常规临时支护 | 平衡的项目默认值 |
| 三级基坑 / 简单场地 | 低复杂度临时工程 | 较低附加储备，仍需项目确认 |

模板值属于项目启动草案，勘察报告、专项审查、地方标准和审图意见优先。

### 3.2 作用分组合同

设计基准新增显式作用分组：

- 土压力；
- 水压力；
- 坑边堆载；
- 车辆荷载；
- 邻近建构筑物附加作用；
- 施工临时荷载；
- 温度作用；
- 预加轴力与安装偏差。

各作用记录启用状态、来源和复核要求。计算内核已经对坑边堆载和水压力启停进行联动；车辆、邻近建筑等专项作用若启用但缺少模型输入，会明确进入 `missing_input` 或人工复核状态。

### 3.3 项目级安全目标

安全目标按工程等级初始化，并允许项目覆盖。目录包括：

- 构件强度；
- 刚度与使用性能；
- 支撑稳定；
- 立柱稳定；
- 嵌固稳定；
- 坑底抗隆起；
- 渗流稳定；
- 承压水突涌；
- 整体稳定。

检查结果同时显示设计值、限值、安全系数、利用率、项目目标和证据状态。

### 3.4 A/B/C Pareto 比选

方案比较新增五类目标：

1. 最大墙体位移；
2. 最大支撑轴力；
3. 最大围檩弯矩；
4. 材料代理指标；
5. 施工复杂度风险。

系统执行非支配排序并输出：

- `paretoRank`；
- `paretoFront`；
- 材料指标；
- 施工复杂度指标；
- 综合工程评分。

排序顺序优先满足硬约束，再考虑 Pareto 前沿和综合评分。代理指标只用于候选初筛；正式采用仍以施工阶段完整计算、构造条件和人工复核为准。

### 3.5 型钢支撑目录

新增可版本化的型钢支撑目录：

```text
packages/sections/steel_support_profiles.json
```

当前内置 4 个常用轮廓作为软件目录示例，支持按所需面积推荐候选截面。正式项目必须使用供应商、企业标准或经审定的项目型钢库确认：

- 截面面积；
- 惯性矩；
- 回转半径；
- 局部稳定参数；
- 钢材牌号；
- 连接构造。

### 3.6 配筋可施工性

配筋深化新增：

- 墙体和支撑钢筋拥挤度；
- 纵筋净距；
- 大直径钢筋机械连接建议；
- 锚固长度检查；
- 搭接长度和错开要求；
- 构件级问题定位；
- 可施工性摘要。

配筋结果区分“承载力满足”和“施工构造可实施”，避免计算配筋通过后直接进入施工图。

## 4. P2：计算模型深化

### 4.1 工程空间分析合同

项目设计基准可以选择：

- `compact_spatial`：核心快速空间模型；
- `engineering_spatial`：考虑开裂刚度、节点半刚性、刚域和初始缺陷的工程空间模型。

每次计算结果保存 `analysisModelContract`，明确：

- 墙体开裂刚度系数；
- 围檩开裂刚度系数；
- 节点平动和转动刚度系数；
- 刚域长度系数；
- 初始缺陷比；
- 长期效应开关；
- 不利情景开关。

### 4.2 开裂刚度和长期效应

墙体和围檩刚度可以采用项目级折减系数。兼作永久结构且启用长期效应时，系统根据持续荷载比例、徐变系数和收缩应变形成透明的长期刚度/位移合同。

当前实现用于统一杆系和墙体工程分析；复杂开裂演化、材料非线性和施工龄期耦合仍需专项有限元复核。

### 4.3 节点半刚性和刚域

全局空间体系增加：

- 墙—围檩连接平动刚度折减；
- 节点转动刚度折减；
- 围檩与支撑交汇刚域；
- 模型假设随结果保存。

该模型用于反映理想铰接与完全刚接之间的工程连接状态，不能替代经试验或精细节点模型标定的连接刚度。

### 4.4 初始缺陷

支撑和立柱空间分析引入项目级初始缺陷比，用于形成稳定计算的等效附加效应。结果中记录缺陷取值和适用构件。

### 4.5 不利情景筛查

新增以下情景：

- 降水失效 / 坑内水位回升；
- 超挖；
- 局部渗流通道；
- 承压水头抬升；
- 永久阶段长期效应。

筛查基于当前施工阶段计算结果进行透明放大或安全系数折减，用于识别专项复算需求。它不冒充第二次非线性求解。进入控制区间的情景必须按不利水位、超挖标高或渗流模型重新组装作用并计算。

### 4.6 局部节点子模型筛查

支撑—围檩节点新增局部检查：

- 承压面积；
- 局部承压利用率；
- 爆裂/劈裂风险代理指标；
- 节点控制轴力；
- 节点编号和位置；
- `screening_implemented` 证据状态。

该检查用于识别需要节点放大和精细分析的位置。对于大轴力、复杂牛腿、钢—混组合、预埋件和多向汇交节点，仍应建立壳/实体局部模型并完成构造审查。

## 5. 数据和界面结果

核心工作区新增但保持有界的数据包括：

- 设计基准模板和作用分组摘要；
- 规范校核证据覆盖；
- A/B/C Pareto 排名；
- 不利情景摘要；
- 型钢目录版本；
- 配筋可施工性摘要。

完整施工阶段数组、矩阵、节点明细和逐根钢筋继续保存在外部对象或按需结果中，不进入普通页面首屏。

## 7. 当前边界和未夸大的能力

- 型钢目录为软件示例目录，正式工程需导入项目或企业截面库；
- 局部节点子模型目前为透明工程筛查，不是认证的非线性壳/实体有限元结果；
- 不利情景目前用于筛查，控制情景必须触发专门施工阶段复算；
- 节点半刚性系数需要试验、构造或项目经验标定；
- 邻近建筑、车辆、地震和特殊施工荷载若缺少专项输入，保持缺资料状态；
- 当前执行环境未安装 `ezdxf`，完整 DXF 压力测试未纳入本版本通过数量；
- Windows 原生大型工程仍建议使用真实项目运行诊断脚本复测内存和 Job Object 限额。

## 8. 升级建议

1. 备份 `runtime/pitguard.sqlite3`、`runtime/artifacts` 和 `exports`；
2. 替换程序文件；
3. 先执行设计基准迁移的 `--dry-run`；
4. 启动系统后逐项目确认设计基准；
5. 对现有工程运行 P0–P2 审计；
6. 重新计算已修改设计基准、分析模型或作用分组的项目；
7. 对进入控制区间的不利工况执行专项复算。
