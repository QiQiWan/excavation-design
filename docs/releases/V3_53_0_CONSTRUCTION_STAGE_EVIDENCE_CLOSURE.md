# PitGuard V3.53.0 施工阶段数据与计算证据闭环

## 1. 结论

页面提示“计算结果缺少施工阶段数据”并不等同于项目真的缺少施工阶段输入。当前案例中，计算页已经显示最大位移、支撑轴力和墙体弯矩，项目栏同时显示工作区与外部成果分别占用空间，说明求解结果曾经生成并被外置保存。旧工作区投影为了控制网页内存，固定把 `stageResults` 置空；核心状态和配筋入口随后只检查这个空数组，因而把“外部成果尚未载入”误报成“施工阶段设计资料缺失”。

V3.53 将施工阶段分成两类数据并分别治理：

- 设计输入：开挖顺序、支撑安装/退出、水位、超载和换撑条件，可在计算页编辑、校验和锁定；
- 计算输出：逐墙、逐阶段内力、位移、支撑轴力和校核记录，只能由完整计算生成，并按结果 ID 从外部成果读取。

二者不再使用同一个“缺资料”提示，也不允许把部分成果当成完整包络进入配筋。

## 2. 证据与根因

| 观察 | 旧系统行为 | 根因 |
|---|---|---|
| 页面已有 3.34 mm、3993.4 kN、313.27 kN·m | 配筋入口仍提示缺少施工阶段结果 | 汇总值保留在工作区，逐阶段数组被外置后置空 |
| 页面显示“工作区 2.7 MB、外部成果 3.0 MB” | 门禁只读取工作区模型 | 外部成果引用存在，但状态接口没有按结果 ID 回载 |
| `verification-coverage.jsonl` 连续记录 `calculationCurrent=false`、全部 `missing_input` | 已计算项目仍显示整表缺资料 | 验算服务接收到空的 `stageResults`，无法映射逐墙、逐阶段证据 |
| 历史任务可成功写入汇总 | 后续页仍显示结果过期 | 成功标记只清除了嵌套状态，顶层 `requiresRecalculation` 可能残留 |
| 旧版完整计算每次重新生成默认工况 | 施工组织确认的换撑顺序可能消失 | 工况没有来源、锁定和修订合同 |

直接故障点位于原工作区压缩函数：无条件输出 `stageResults=[]`，且没有保留阶段实际数、期望数和存储状态。后续 `calculation_readiness` 将数组为空解释为未生成，而不是外部化。修复同时覆盖存储、仓储、门禁、计算任务和前端，避免只在页面上改文案。

## 3. 哪些资料在哪里补齐

| 数据 | 补齐位置 | 责任方 | 设计阶段可提供 | 操作 |
|---|---|---|---|---|
| 基坑顶、底标高和闭合轮廓 | 工程输入 → 基坑轮廓 | 基坑设计 | 是 | 先确认设计坑底，阶段标高必须位于坑顶与坑底之间 |
| 阶段开挖标高和顺序 | 计算验算 → 施工阶段编辑器 | 基坑设计、施工组织 | 是 | 按工作面逐级下降，最终阶段必须覆盖设计坑底 |
| 支撑构件及标高 | 围护方案 | 支护结构设计 | 是 | 先采用当前围护方案，再在阶段中选择当前真实支撑 ID |
| 支撑安装并形成传力的阶段 | 计算验算 → 施工阶段编辑器 → 已激活支撑 | 支护结构设计 | 是 | 工作面开挖到支撑标高以下后再激活；列表应包含此前仍工作的支撑 |
| 换撑或拆撑对象 | 施工阶段编辑器 → 本阶段退出支撑 | 支护设计、施工专项 | 是 | 只能退出此前已经激活的构件 |
| 换撑/拆撑生效条件 | 施工阶段编辑器 → 换撑/拆撑生效条件 | 地下结构设计、施工专项 | 是 | 写明楼板强度、连接完成、传力转换和验收条件 |
| 坑外设计水位 | 项目设置 → 基本设计控制 | 勘察、岩土设计 | 是 | 采用勘察与设计控制水位；分期变化可在阶段中覆盖 |
| 坑内控制水位 | 项目设置或施工阶段编辑器 | 降水设计 | 是 | 按降水分期录入坑内控制水位 |
| 阶段地面超载 | 项目设置或施工阶段编辑器 | 总图、基坑设计 | 是 | 填写坑边堆载、交通与施工荷载，不得为负值 |
| 逐阶段内力、位移和轴力 | 计算验算 → 计算当前方案 | 软件计算生成 | 不适用 | 不能手工补录；确认上述输入后重新运行完整计算 |
| 外部阶段成果分块 | 项目成果存储 | 系统维护 | 不适用 | 正常情况自动读取；分块缺失或校验失败时恢复 artifacts 备份，无法恢复则复算 |

页面中的“这些资料在哪里补齐”会返回同一份结构化清单。设计人员只需要录入前九类输入；最后两类属于计算输出和系统成果，不应要求设计人员重复填写。

## 4. 修复实现

### 4.1 结果摘要和选择性回载

每份结果新增 `stageResultSummary`：

- `actualCount`：已持久化阶段记录数；
- `expectedCount`：计算质量包声明的期望数；
- `chunkCount`、`storedBytes`：外部成果分块信息；
- `storageState`、`loadState`、`complete`：存储和读取完整性。

最新证据接口只加载当前结果的阶段分块和结果明细，不加载完整地质网格、A/B/C 历史计算、监测历史及逐根配筋制造数据。读取型服务继续保持有界内存；写入型服务从规范项目快照开始，避免把下采样工作区保存回权威项目。

### 4.2 五态诊断

| 状态 | 含义 | 用户动作 | 配筋门禁 |
|---|---|---|---|
| `not_generated` | 没有生成逐阶段结果 | 检查阶段后运行完整计算 | 阻断 |
| `external_unloaded` | 成果已外置，当前调用未加载 | 系统按需加载，无需补设计资料 | 阻断当前调用 |
| `loaded` / `inline` | 权威阶段成果完整 | 可继续检查计算合同与硬失败 | 允许进入下一项门禁 |
| `partial` | 只读取到部分分块或数量不足 | 恢复成果或重新计算 | 严格阻断 |
| `artifact_missing` | 索引声明有成果，但文件缺失/损坏 | 检查成果存储和备份；否则复算 | 严格阻断 |

只有完整阶段证据、当前计算合同、零硬失败且计算质量包有效时，配筋入口才通过。

### 4.3 施工阶段编辑与保护

施工阶段现在记录 `source`、`locked`、`revision` 和更新时间。保存时执行：

- 阶段 ID 唯一；
- 标高位于坑顶/坑底范围并按顺序单调下降；
- 最终阶段到达设计坑底；
- 支撑 ID 属于当前拓扑；
- 支撑不得在暴露前激活；
- 支撑不得未激活先拆除，也不得同阶段同时激活和退出；
- 换撑/拆撑必须选择退出对象并提示补充生效条件；
- 水位有限、超载不小于零；
- 工况拓扑哈希与当前采用方案一致。

用户保存的工况会锁定。完整计算优先使用该工况；旧版“仅生成工况”动作发现锁定工况时直接保留，只有明确点击“恢复推荐阶段”才会重建默认值。修改阶段会使旧计算失效，成功复算后同时清除两层失效标记。

## 5. 升级后的操作顺序

1. 进入“计算验算”，展开施工阶段编辑器；
2. 检查最终坑底、支撑激活/退出、水位、超载和换撑条件；
3. 保存并锁定项目阶段；
4. 运行“计算当前方案”；
5. 查看“施工阶段计算证据”，确认状态为 `loaded` 或 `inline`，已载入数等于期望数；
6. 再进入配筋深化，生成并应用正式配筋；
7. 如显示 `partial` 或 `artifact_missing`，不要补录内力，先恢复外部成果或重新计算。

升级会更新计算算法与规则合同。旧结果仍可查看摘要，但为了建立新的阶段完整性证据，项目应重新执行一次当前方案完整计算。
