# PitGuard V3.48.0：候选方案与支撑配筋完整性修复

## 1. 版本目标

V3.48.0 集中解决两个反复影响工程使用的问题：A/B/C 候选方案显示与实际几何不一致，以及水平支撑三维钢筋只显示纵筋。版本同时修复配筋截面调整后的计算闭环、计算追溯接口对结构化证据不兼容、以及诊断日志不足等问题。

## 2. 日志与现状结论

用户上传的后端日志来自 V3.39、V3.42 等较早运行实例，并非 V3.47 的 `runtime/diagnostics`。这些日志仍能证明历史链路存在以下风险：

- 重任务提交后长时间只轮询任务状态，没有终止状态和阶段证据；
- 渐进式配置发生多次 409 冲突；
- `diagnose-and-repair` 同步请求耗时超过 12 秒；
- `calculation/trace` 因 `demandValue` 为字典而返回 500；
- 旧版项目摘要无法接受 `workspace_only` 状态。

`workspace_only` 和隔离 worker 已在前序版本修复。V3.48 继续修复计算追溯的结构化证据兼容，并补充候选几何、配筋契约和可视化交付日志。

## 3. A/B/C 方案比选修复

### 3.1 上层卡片改用真实平面几何

删除将长条形基坑的多层水平支撑投影成“墙片”的伪三维缩略图。上层 A/B/C 卡片与下层详图统一使用真实基坑平面、围护轮廓、主对撑、角撑、次撑、立柱和节点数据。

三张卡片共享同一包围盒和 `viewBox`：

- 几何中心一致；
- 缩放比例一致；
- 四周留白一致；
- 长条形基坑横向居中；
- 不再分别缩放造成视觉误判。

### 3.2 真实几何去重

候选指纹从“仅主对撑”扩展到全部支撑角色：

- 主对撑；
- 角部斜撑；
- 次撑；
- 环撑/径向撑；
- 换撑构件。

指纹包含支撑层、角色和规范化端点。体系名称、位置策略名称或小幅元数据变化不能形成独立候选。实际端部斜撑、支撑数量、立柱数量或分仓发生变化时才能进入 A/B/C。

### 3.3 受控阻断候选

当所有方案均未通过硬约束时，系统最多保留三套实际几何不同的诊断方案，用于比较支撑密度、端部处理和立柱服务跨。诊断方案明确标记：

- `capabilityOutcome=controlled_block`；
- `formalSchemeEligible=false`；
- 禁止采用；
- 禁止完整计算；
- 禁止正式出图。

这样既避免伪造三个“可用方案”，也不会只显示一张难以分析的失败卡片。

### 3.4 代理值与正式计算分离

未执行完整施工阶段计算时，仅显示：

- 几何与拓扑指标；
- 无量纲轴力代理；
- 待计算的位移、围檩内力和安全系数状态。

正式设计值只来自 `candidate_comparison` 任务。

## 4. 水平支撑配筋修复

### 4.1 支撑钢筋契约

每根钢筋混凝土水平支撑应至少具有：

1. `longitudinal`：纵向主筋；
2. `distribution`：侧面分布筋/架立筋；
3. `stirrup`：端部和跨中箍筋；
4. `tie`：拉结筋；
5. `additional`：节点、吊装、搭接或局部附加筋。

三维接口新增逐支撑 `supportContracts`，同时返回已存在类型、由正式配筋表恢复的类型、缺失类型和完整性状态。

### 4.2 旧项目恢复

旧项目中常见的状态是 `support.reinforcement` 只保留纵筋，而完整箍筋配置仍位于 `rebarDesignScheme.supportSchemes`。V3.48 在三维可视化前按构件 ID/编号恢复缺失的分布筋、箍筋、拉结筋和附加筋。

若正式配筋表也不存在，接口明确返回 `incomplete` 和缺失类型，不再静默显示成完整钢筋笼。

### 4.3 可视化采样和近景模式

逐根钢筋仍采用有界抽样，但每个支撑和全局场景均为五类钢筋预留配额，避免纵筋先耗尽全部预算。

新增“支撑构造近景”：

- 自动定位代表性水平支撑；
- 隔离围护墙和其他构件；
- 放大钢筋直径显示尺度；
- 保持纵筋、箍筋、分布筋、拉结筋和附加筋显隐开关；
- 可返回全局模型。

### 4.4 配筋—重算闭环

配筋策略可能增大支撑或围檩截面。旧流程在应用截面后立即使计算失效，同时前端又以 `recalculate=false` 提交任务，形成“深化配筋完成即不可用”的死路。

V3.48 改为一次有界闭环：

```text
第一次配筋
→ 发现截面变化
→ 保存新截面
→ 完整施工阶段重算
→ 按新内力包络第二次配筋
→ 保存正式配筋摘要
```

最多执行一次自动重算。第二次仍需截面变化时进入人工复核，不循环计算。

## 5. 计算追溯与运行日志

`calculation_trace` 现在可读取数值、字符串、列表或结构化字典。字典中的 `value`、`designValue`、`calculatedValue`、`maximum`、`factor` 等字段会提取为标量，原结构化证据同时保留在 `demandEvidence`/`capacityEvidence`。

新增日志：

- `candidate-geometry.jsonl`；
- `rebar-contract.jsonl`；
- `rebar-visualization.jsonl`；
- `rebar-task.jsonl`。

新增审计：

```bash
python scripts/audit-design-rebar-integrity.py --project-id <PROJECT_ID>
```

## 6. 230 m 台阶形算例结果

严格几何去重和受控阻断逻辑下：

- 试算上限：12；
- 诊断候选：3 套实际几何不同方案；
- 当前采用体系支撑：208 根；
- 立柱：40 根；
- 候选生成与写回：约 17.1 s；
- 峰值 RSS：约 411 MB；
- 诊断候选未写入当前正式体系；
- 完整项目快照：约 1.30 MB；
- 网页工作区：约 0.50 MB。

该算例的三套方案属于受控阻断诊断方案，不能作为正式 A/B/C 设计方案。该结论由硬约束状态决定，系统不再使用不同标签伪装为三个有效方案。

## 7. 当前边界

- 当前版本能保证候选“真实不同”，不能保证任意复杂平面必然产生三套通过硬约束的轴压杆体系；必要时应切换环梁、中心岛、分区或显式框架体系。
- 旧项目只有纵筋且无正式支撑配筋表时，系统只能报告缺失，不能凭空生成正式构造参数。
- 3D 预览是有界抽样，完整数量和加工信息仍以钢筋表、BBS、CSV/XLSX 和深化成果包为准。
- 用户上传日志未包含 V3.47 的 `candidate-geometry`、`rebar-contract`、`rebar-visualization` 和 `rebar-task`，升级后应重新运行并提交这些文件进行项目级复核。
