# PitGuard V3.49.0：设计基准静默交互与台阶形支撑闭合修复

## 1. 版本目标

V3.49.0 针对丰收湖项目的当前运行结果完成四项收口：

1. 将“参数影响”改为与设计基准控件联动的紧凑决策侧栏；
2. 任务或阶段完成后保持当前页面，不再主动显示横向完成说明；
3. 修复台阶形、凹角形轮廓中支撑端点误挂接到切向回折墙的问题；
4. 补充候选搜索预算、任务失败原因和 Windows 进程内存诊断。

## 2. 日志结论

当前 API 启动、认证、项目读取、工作区保存和任务提交均能返回 HTTP 200。日志中的 `WinError 10054` 为浏览器/连接端提前断开产生的单次连接重置，服务随后继续响应，不是围护结构计算失败的根因。

任务存在大量轮询，但旧任务生命周期日志没有保存异常类型和 traceback 摘要；Windows worker 内存日志还出现 RSS/Private Bytes 全为 0 的情况。这导致任务失败和内存异常难以定位。诊断压缩包中两次 `calculation_full` 结束为 failed，但旧 `task-finish` 没有错误原因；当前支撑优化记录为 `controlledBlock=true`、`selectedCandidateId=null`，写回时仍保留 168 根支撑和 44 根立柱。候选进度文本按 48 组显示，实际诊断记录完成约 30 组试算，说明前端请求、运行上限和日志表达没有统一。

前端 Vite 在 127 ms 内正常启动，未发现编译或启动错误。当前故障集中在工程拓扑、任务证据和 Windows 资源监测层。

## 3. 丰收湖支撑闭合根因

丰收湖轮廓在 `x=-39 m` 等台阶凹角同时存在水平墙段和竖向回折墙。旧端点匹配仅按点到墙段距离选择最近墙面。支撑沿 Y 向传力时，端点可能被挂接到与支撑轴线近似平行的竖向回折墙。

该模型在平面图上接近闭合，但轴压杆端部没有形成垂直于传力方向的墙—围檩承压面，因此设计资格门禁判定为“支撑体系与传力拓扑未闭合”。

V3.49 新增方向感知的墙面选择：

- 根据支撑轴线计算传力方向；
- 计算候选墙段内法线与支撑轴线的对齐度；
- 优先选择具有有效法向承压关系的墙段；
- 拒绝仅距离接近但与支撑相切的回折墙；
- 同步更新墙面编号、真实承压点、中心线净距、支撑跨度和墙节点。

修复只调整端点的墙面语义和承压位置，不新增虚构支撑，也不绕过拓扑、跨度、交叉或稳定性检查。

## 4. 旧项目自动恢复

正式计算门禁前会运行一次幂等修复：

```text
读取当前支撑
→ 识别切向墙面挂接
→ 按传力方向选择有效承压墙面
→ 更新支撑和节点
→ 失效旧计算工况与结果
→ 重新执行完整设计资格检查
```

首次运行可能改变旧项目；第二次运行不会重复修改。另提供人工审计脚本：

```bash
python scripts/repair-v349-support-bearings.py --project-id <PROJECT_ID>
python scripts/repair-v349-support-bearings.py --project-id <PROJECT_ID> --apply
```

默认是 dry-run。`--apply` 保存修复并清除已失效的旧计算结果。

## 5. 设计基准交互调整

原“选项会影响什么”长文本改为右侧紧凑决策侧栏：

- 工程等级与安全目标；
- 场地与周边环境；
- 荷载与规范组合；
- 结构分析模型；
- 材料与构造参数；
- 企业工程资源。

每一组显示当前取值和受影响的计算输出。点击影响组会定位并聚焦对应控件；聚焦或修改控件时，右侧自动切换到相应影响组。页面宽度不足时，侧栏自动移到表单下方。

## 6. 静默完成行为

移除所有阶段完成后的全局“修改说明”和横向长句。以下操作完成后均停留当前阶段：

- 设计基准确认；
- 地质模型建立；
- A/B/C 生成和方案采用；
- 当前方案计算；
- 配筋深化；
- 不利工况和 P3 深化；
- 成果导出。

系统只更新步骤状态、按钮可用性和结果数据。用户通过顶部六阶段导航自行进入下一步。

## 7. 搜索与任务诊断增强

### 7.1 候选搜索预算

前端或历史配置即使请求 48 组试算，Core 模式仍受运行时硬上限控制：

```text
PITGUARD_SUPPORT_CANDIDATE_TRIAL_LIMIT=12
```

诊断日志增加 `search-budget-resolved`，记录请求值、产品模式、运行上限和实际试算数。

### 7.2 任务失败证据

`task-lifecycle.jsonl` 新增：

- `task-error` 事件；
- `errorType`；
- `errorMessage`；
- `tracebackTail`；
- `task-finish` 中的最终错误字段。

### 7.3 Windows 内存采集

修复 64 位 Windows 中 `GetCurrentProcess` 返回值被 ctypes 默认 `c_int` 截断的问题。内存快照增加：

- `processMetricsAvailable`；
- `processMetricsSource`；
- RSS；
- Peak RSS；
- Private Bytes；
- Pagefile Usage；
- 有效进程内存。

原生 API 不可用时使用 `psutil` 回退。该代码在当前 Linux 环境完成编译和接口测试，仍需在用户的 Windows 工作站确认原生计数值。

## 9. 升级后的建议操作

1. 保留 `runtime/pitguard.sqlite3`、`runtime/artifacts` 和 `exports`；
2. 完全关闭旧 API、worker、Vite 和遗留 Python 进程；
3. 使用 `start-windows.bat` 或 `start-windows.ps1` 启动；
4. 打开丰收湖项目后重新生成 A/B/C；
5. 采用正式候选并重新运行当前方案计算；
6. 检查 `runtime/diagnostics/task-lifecycle.jsonl`、`candidate-search.jsonl` 和 `worker-memory.jsonl`。

旧项目需要先核查修复范围时，可运行 `repair-v349-support-bearings.py` 的 dry-run。
