# PitGuard V3.32.0 即时交互与全局进度反馈

## 1. 问题定位

V3.31 已完成冷热数据分层，但交互仍存在两类问题：

1. 多个轻量接口串行请求，用户需要等待但页面没有统一状态；
2. `SQLiteProjectStore` 在每个 HTTP 请求创建仓储时重复执行 schema 检查、JSON1 回填和统计更新，形成不必要的磁盘写入、数据库锁和延迟。

第二项是项目列表、创建、打开和保存普遍偏慢的重要根因。

## 2. 核心改造

### 2.1 轻量请求即时化

- 登录启动由 `auth/status + auth/me` 两次串行请求合并为 `auth/bootstrap` 一次请求；
- 项目列表优先显示 sessionStorage 中的摘要，后台执行短周期刷新；
- 创建项目立即插入临时行，服务器成功后替换；
- 删除项目立即从列表移除，失败时自动回滚；
- 普通项目保存使用 `PATCH /api/projects/{id}/workspace`，服务端保存后直接返回受控工作区 JSON；
- 设计参数保存不再额外执行一次项目 GET。

### 2.2 全局进度反馈

所有通过统一 API client 发起的请求均自动产生：

- 260 ms 延迟显示的顶部进度条；
- 操作名称、当前阶段、估算百分比；
- 成功/失败终态；
- 写操作透明交互屏障，阻止重复点击；
- 同一在途请求自动去重。

后台任务轮询属于安静请求，不会造成进度条闪烁；后台任务仍使用服务器返回的真实进度。

### 2.3 数据库请求路径优化

`SQLiteProjectStore` 的 schema 初始化改为：

```text
同一 API 进程 + 同一数据库路径
→ 只执行一次 schema/迁移检查
→ 后续请求只建立短生命周期 SQLite 连接
```

仓储依赖复用同一已初始化 store 配置，但不共享 SQLite connection，因此保持线程安全。

本地基准中，完成首次 schema 初始化后，连续 100 次“创建 store + 查询项目摘要”总耗时约 72.8 ms，平均约 0.73 ms/次。

### 2.4 可观测性

每个响应新增：

- `X-PitGuard-Request-Id`；
- `X-PitGuard-Duration-Ms`；
- `Server-Timing: app;dur=...`。

超过 `PITGUARD_SLOW_REQUEST_MS` 的请求写入 `pitguard.performance` 日志，默认阈值 1200 ms。

## 3. 运行策略

- 小于约 300 ms 的操作通常不显示进度条，保持界面干净；
- 超过约 300 ms 的读取显示非阻塞进度；
- 写入、删除、登录和任务提交显示阻塞式进度，避免误操作；
- 长计算、IFC、图纸和钢筋导出继续由隔离 worker 执行并显示真实阶段进度。

## 4. 兼容性

前端优先调用 `/api/auth/bootstrap`。若滚动升级过程中连接到旧 API 节点，会自动回退到 `/api/auth/status` 和 `/api/auth/me`。

## 5. 验证

- 前端 12 个测试文件、20 项测试通过；
- V3.29—V3.32 存储、韧性与交互专项 23 项通过；
- TypeScript 和 Vite 生产构建通过；
- Python compileall 通过；
- 新增 schema 单次初始化、workspace 快速保存、认证 bootstrap、摘要读取测试。

## 6. 本地轻量路径基准

在 30 个项目的临时 SQLite 数据库上，使用 FastAPI TestClient 测得中位数：

| 操作 | 中位耗时 | 最大值 |
|---|---:|---:|
| 认证 bootstrap | 3.20 ms | 36.04 ms |
| 项目列表 | 6.13 ms | 8.76 ms |
| 工作区打开 | 3.88 ms | 6.44 ms |
| 工作区增量保存 | 6.56 ms | 18.71 ms |

该结果用于验证服务端代码路径，未包含公网延迟、TLS、Nginx、磁盘繁忙和真实大型项目差异。
