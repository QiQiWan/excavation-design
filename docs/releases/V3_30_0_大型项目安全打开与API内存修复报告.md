# PitGuard V3.30.0 大型项目安全打开与 API 内存修复报告

## 1. 现场故障判断

现场状态中，`pitguard-worker` 处于空闲状态，RSS 约 51 MB；`pitguard-api` 的 `MemoryCurrent` 约 2.23 GB，已经超过 `MemoryHigh=2 GB`，且该服务的 `MemoryMax` 仅为 4 GB。项目列表、登录、诊断和健康检查可以返回 200，但日志没有出现单个项目详情请求成功完成的记录。

这组现象说明，当前故障主要发生在“打开项目”的 API 数据加载链路，而非隔离计算 worker。现有日志没有直接给出 `Killed process`、`oom-kill` 或退出码，因此不能仅凭该片段断言每次都由 Linux OOM Killer 杀死；但 API 已越过内存高水位，打开大型项目时继续产生数倍内存放大，触发 cgroup `MemoryMax`、内核 OOM 或长时间内存回收的风险很高。

## 2. 原有项目打开链路的内存放大

V3.29.0 及以前的项目读取流程为：

```text
SQLite projects.data 完整 JSON 字符串
→ sqlite3 将完整字符串复制到 Python
→ json.loads 构造完整 dict/list 对象树
→ Project.model_validate 构造 Pydantic 对象树
→ 旧版本迁移、历史结果检查和压缩
→ model_dump/model_dump_json 再构造响应对象
→ FastAPI response_model 再验证并序列化
→ 浏览器一次性接收全部项目数据
```

一个大型项目可能同时包含：

- 全施工阶段墙体离散位移、弯矩和剪力；
- 全局耦合自由度、支撑反力和数值诊断；
- A/B/C 候选方案的完整计算结果；
- 候选差异动画与完整平面几何；
- 原始 VTU 网格和地质表面；
- 逐根钢筋几何、加工表和碰撞结果；
- IFC、图纸、工业资格与深化设计缓存；
- 多个历史计算结果及不可变修订。

磁盘上的 JSON、Python 字符串、字典/列表、Pydantic 对象、响应字典和最终 JSON 字节可能同时存在。真实工程对象包含大量小字典和列表时，峰值 RSS 往往达到原始 JSON 文件大小的数倍。旧读取操作还可能自动执行项目迁移并写入新修订，进一步增加一次完整序列化和数据库副本。

## 3. V3.30.0 存储模型

`projects` 表增加：

```text
workspace_data   受控工作区投影
payload_bytes    完整项目快照字节数
workspace_bytes  工作区投影字节数
```

完整数据仍保存在：

```text
projects.data
project_revisions.data
```

不会丢失计算结果、钢筋数据或历史修订。网页打开项目默认只读取 `workspace_data`；完整快照只允许隔离 worker、维护进程或显式的受控全量接口读取。

## 4. 工作区投影内容

工作区保留：

- 项目基本信息、单位和坐标；
- 钻孔、土层和关键地质摘要；
- 开挖轮廓、施工障碍和周边约束；
- 围护墙、围檩、支撑、立柱及当前采用方案；
- 支撑候选指标和有限预览；
- 施工阶段；
- 最近一次计算的控制值、校核汇总、质量状态和发行状态；
- 审批、图纸和修订摘要。

工作区排除或压缩：

- 全部阶段结果矩阵和高密度离散点；
- 原始 VTU 网格；
- 候选完整计算；
- 大型深化与 IFC 实体缓存；
- 逐根钢筋完整几何；
- 历史计算结果全文；
- 过量监测时序。

地质表面最大降采样到 64×64。普通工作区默认上限为 24 MB；超过上限时继续移除候选预览几何、地质面网格和非必要监测记录，保留可编辑工程对象与关键指标。

## 5. 零反序列化打开路径

默认接口调整为：

```text
GET /api/projects/{project_id}?profile=workspace
```

返回链路为：

```text
SQLite workspace_data
→ Starlette/FastAPI 原始 Response
→ 浏览器
```

该路径不执行 `json.loads`、完整 Pydantic 项目构造、历史迁移、项目写回和 `response_model` 二次验证。响应头同时给出项目修订号、完整快照字节数和工作区字节数。

## 6. API 全量读取硬门禁

生产 API 默认配置：

```text
PITGUARD_PROCESS_ROLE=api
PITGUARD_API_FULL_PROJECT_LIMIT_MB=96
PITGUARD_MIGRATE_ON_FULL_LOAD=0
```

当完整快照超过 96 MB 时，API 在读取 `projects.data` 之前返回 HTTP 413：

```text
PROJECT_FULL_LOAD_BLOCKED
```

不会把大字符串加载到 API 进程。worker 设置 `PITGUARD_PROCESS_ROLE=worker`，仍可读取完整快照执行计算、IFC、图纸和钢筋交付。

## 7. 读取操作改为无副作用

以下操作不再触发完整项目迁移或保存：

- 打开项目；
- 查询项目修订号；
- 查询修订列表；
- 查询审计记录；
- 查询工作区仪表板。

旧模型迁移、历史压缩和算法版本升级由隔离 worker 或明确维护任务执行。GET 请求不再隐式生成新修订。

## 8. 旧数据库在线迁移

首次执行 `start-linux.sh` 时：

1. 使用 SQLite 在线备份 API 创建升级前一致性备份；
2. 增加 `workspace_data`、`payload_bytes` 和 `workspace_bytes`；
3. 使用 SQLite JSON1 在数据库侧生成旧项目的工作区投影；
4. 移除工作区中的计算全文、VTU、逐根钢筋和候选完整计算；
5. 计算完整快照和工作区字节数；
6. 输出体积最大的项目；
7. 启动 API 与 worker。

迁移在短生命周期维护进程中执行。即使旧数据库很大，迁移进程结束后其内存会由操作系统完整回收，不会污染常驻 API。

## 9. 前端修复

- 项目打开固定请求 `profile=workspace`；
- 打开按钮显示“安全加载中…”；
- 项目列表显示完整快照体积和存储状态；
- 20 秒超时后显示存储诊断提示；
- 后台任务完成后刷新工作区投影；
- 重型计算结果继续按需读取，不随项目首页一次加载；
- 打开大型项目不会在浏览器端构造无关的完整历史对象。

## 10. 运维诊断

新增：

```text
GET /api/projects/{project_id}/storage-health
```

返回完整快照体积、工作区体积、压缩比例、当前进程角色、API 全量读取限制及是否允许全量读取。

`status-production.sh` 会列出体积最大的十个项目：

```bash
bash status-production.sh
```

也可直接查询数据库：

```bash
sqlite3 /opt/excavation-design/runtime/pitguard.sqlite3 \
  "select name, round(payload_bytes/1048576.0,2), round(workspace_bytes/1048576.0,2) from projects order by payload_bytes desc limit 20;"
```

## 11. 验证结果

- V3.30 专项测试：7 项通过；
- V3.20—V3.29 关键后端回归：66 项通过；
- 前端 12 个测试文件、20 项测试通过；
- TypeScript 编译与 Vite 生产构建通过；
- Python `compileall` 通过；
- Shell 脚本语法检查通过。

64 MB 合成项目基准中：

- 工作区 JSON 约 4 KB；
- 工作区读取进程最大 RSS 约 295 MB；
- 旧式完整 Pydantic 加载与响应最大 RSS 约 555 MB；
- 测试环境 Python 基线约 280 MB。

该合成数据主要由大字符串构成。真实工程项目包含大量小对象，完整 Pydantic 对象树通常具有更高的内存放大系数。

## 12. 部署步骤

V3.30.0 修改了数据库结构、API 项目读取方式、前端和 systemd 环境，首次升级必须执行：

```bash
cd /opt/excavation-design
sudo bash start-linux.sh
```

不能只运行 `restart-production.sh`。

部署后检查：

```bash
bash status-production.sh
curl -fsS http://127.0.0.1:8002/health/ready
sudo journalctl -u pitguard-api -f
```

打开项目时，浏览器请求应为：

```text
/api/projects/<project_id>?profile=workspace
```

## 13. 若升级后仍被终止

需补充检查内核和 cgroup 记录：

```bash
sudo journalctl -u pitguard-api --since "30 minutes ago" --no-pager
sudo journalctl -k --since "30 minutes ago" \
  | grep -Ei "oom|out of memory|killed process|memory cgroup"
systemctl show pitguard-api \
  -p Result -p ExecMainCode -p ExecMainStatus \
  -p MemoryCurrent -p MemoryPeak -p MemoryHigh -p MemoryMax
```

若存在 OOM 记录，可据此判断是 API cgroup、宿主机内核还是其他服务造成。V3.30.0 会把项目打开路径限制在工作区投影内，剩余全量计算和导出应继续由隔离 worker 执行。

## 14. 后续工业化方向

当单个完整项目长期超过数百 MB，应进一步把单体 JSON 拆分为：

- 项目主数据关系表；
- 地质网格对象存储；
- 计算结果分阶段分表；
- 候选结果独立表；
- 钢筋与 IFC 文件对象存储；
- 修订清单与内容哈希。

V3.30.0 已解决“打开项目即把全部工程成果装入 API 内存”的直接故障，并为后续分表和对象存储迁移建立兼容边界。
