# PitGuard V3.87.7：换撑传力路径自动恢复与可审计闭合

## 1. 诊断结论

本轮诊断包显示，自动闭合没有进入候选计算：任务在施工阶段校验处终止，日志连续报告 52、104、156 根旧支撑引用。该数量与当前每层约 52 根支撑的累计引用一致，说明控制工况保留了旧构件 ID，而支撑拓扑已重新生成。界面中的“已评估 0”因此表示优化器尚未运行，不代表七类优化候选全部失败。

诊断记录还把 7 个施工阶段乘以 20 个墙段形成的 140 条阶段—墙段结果标记为 `stageCount=140`，容易被误读为 140 个施工阶段。V3.87.7 将其拆分为 `stageCount=7` 与 `stageSegmentResultCount=140`。

## 2. 核心修复

### 2.1 构件 ID 与工程语义解耦

`DesignControlStage` 新增必需支撑层、退出支撑层、换撑动作、路径状态和来源。拓扑重建后优先按支撑层语义重绑当前构件 ID，不再把随机生成的构件 ID 当作传力路径本身。

### 2.2 标准换撑序列重建

对于系统生成、未冻结且已声明或可由旧 `replace-Ln` 语义确认的标准自下而上换撑路径，系统自动：

1. 识别当前支撑层；
2. 重建逐层退出顺序；
3. 写入当前支撑 ID；
4. 保留永久结构达到强度、连接完成和传力验收条件；
5. 重新同步正式计算工况。

用户冻结或专项非标准换撑仍要求工程师确认，系统不会擅自改变设计意图。

### 2.3 不再零候选退出

如果正式换撑路径仍需人工确认，优化器继续采用“当前拓扑可审计筛查工况”评估候选，并区分：

- `closed`：结构计算与正式阶段合同均闭合；
- `calculated_pending_transfer_review`：结构筛查闭合，正式换撑路径待确认；
- `cannot_close`：在当前候选和参数边界内无法闭合。

### 2.4 病态支撑层归并

旧数据中按构件高程漂移形成超过 6 个自动支撑层时，系统在未锁定条件下聚类、限幅并按语义层重建支撑，避免工况和搜索规模异常放大。

### 2.5 用户锁定工况优先

显式保存并锁定的用户工况保持最高优先级。语义工况可用于校核和恢复，但不会把用户工况的 `source=user_defined` 改写为同步工况。

## 3. 用户界面

计算页显示自动恢复阶段数、换撑序列是否重建、候选评估数、工况来源和剩余人工项。若仅剩专项换撑确认，界面提供“定位并确认换撑工况”，直接回到施工阶段编辑器。

## 4. 工程边界

自动恢复只处理构件身份重绑和已声明的标准换撑序列。测量坐标、土层参数、地下水、荷载、永久结构方案、冻结阶段和专项非标准传力路径保持人工控制。
