# V3.81.0 升级与数据迁移指南

## 升级前

1. 备份项目数据库、外置对象目录和成果目录；
2. 保留 V3.77 项目 JSON 和已发行成果哈希；
3. 停止 API 与 worker，避免迁移期间写入。

## 自动迁移

首次读取业务工作流或调用 `POST /api/projects/{projectId}/workflow/migrate-legacy-stages` 时：

- 读取最新 `CalculationCase`；
- 将阶段名称、开挖标高、支撑激活、水位和荷载复制为 `DesignControlStage`；
- 创建锁定的“设计控制工况计算”案例；
- 保留原 `CalculationCase` 及历史结果供审计；
- 不把计划日期、审批字段或旧文本解释为现场实测事实。

迁移是幂等操作。已有设计控制工况时不会重复创建；仅在人工确认后使用 `force=true` 重建。

## 人工确认

历史项目应逐项确认：

- 开挖控制标高上下限；
- 每一工况必须激活的支撑；
- 坑内水位与坑边荷载限值；
- 允许超挖和支撑刚度折减；
- 预加轴力目标及允许范围；
- 进入下一工况的控制条件。

施工计划和现场实测应在对应协同模块新建，不能写回设计控制工况。

## 状态迁移

原 `officialIssueReady` 不再同时代表全部业务：

- 设计成果使用 `designIssueGate`；
- 施工准备使用 `constructionPreparationGate`；
- 现场阶段使用 `fieldStageReleaseGate`。

历史设计成果和签署记录保持只读。缺少施工期资料不会作废已发行设计；若新的施工计划或现场状态超出设计允许域，应发起设计复核或新版本设计变更。

## 回滚

升级包不删除旧阶段和旧成果。回滚时恢复数据库和对象目录备份即可。已经创建的 V3.81 协同对象在旧版本中会被忽略，回滚前应额外导出项目 JSON 留档。

## 审批状态更新

迁移后将 `dataStatus` 从 `draft` 调整为 `approved` 或 `frozen`，且其他设计控制字段未变化时，不会使当前计算失效。若控制标高、支撑、水位、荷载、预加轴力或允许偏差发生变化，系统会自动失效当前计算和情景包络并要求复算。
