# PitGuard V3.27.0 多平面支撑拓扑与隔离计算进程

## 1. 版本目标

V3.27.0 同时解决两类相互放大的问题：

1. 长条形基坑端墙围檩支点间距无法闭合，A/B/C 候选重复显示“拓扑 Fail”；
2. 近方形、L/T/U 形和一般凹多边形被任意角度斜撑强行补齐，形成视觉整洁但传力不成立的支撑；
3. 支撑优化和完整计算在 Uvicorn API 进程内运行，候选搜索、深拷贝、密集矩阵和本地数值线程可将整机 CPU 拉满，进而使登录、健康检查和项目保存同时离线。

## 2. 平面形状识别

系统先建立局部主轴坐标系，再计算长短跨、长宽比、凹角数、正交边比例、圆度和径向离散系数。当前分类包括：

- 狭长四边形；
- 近方形四边形；
- 旋转或一般凸四边形；
- 狭长凸多边形；
- 一般凸多边形；
- 正交凹形走廊平面；
- 一般凹多边形；
- 圆形或多边形竖井。

支撑扫描线、转角判定和长宽比全部采用局部主轴，不再依赖全局 X/Y 包围盒，因此旋转长方形可正常生成短跨对撑和平行角撑。

## 3. 长条形端墙闭合

长条形算例的“Fail 6”来源为两端短墙在三道支撑层各有一处围檩跨超限，共计 2×3=6 项。V3.27.0 根据端墙长度和围檩硬限值反算每个转角所需的角撑数量与最大链距，使两组镜像平行角撑在端墙中部留下的无支承跨不超过限值。

默认角撑族允许：

- 每角最多 6 根；
- 最大墙上链距 18 m；
- 最大占用相邻墙长比例 0.55；
- 墙上节点间距 2.5–6.0 m；
- 同组角度离散不超过项目容差。

160 m×33 m 三道支撑资格算例达到：非法穿越 0、支撑中部终止 0、角撑扇形异常 0、角撑节点拥挤 0、围檩跨失败 0。

## 4. 凹形与近方形安全边界

当前结构求解器把普通水平支撑作为两端落墙的轴压杆。对于 L/T/U 形、深凹口和近方形平面，单一方向墙—墙轴压杆往往无法同时闭合所有回墙围檩跨。旧算法通过任意斜撑或支撑中部 T/Y 节点降低几何失败数，实际引入未建模的平面内弯矩和节点反力。

V3.27.0 采用以下规则：

- 凹形平面仅允许连接相互可见且受力方向合理的墙面对；
- 凹形回墙补强优先选择近似平行的对向墙面；
- 支撑若以近切线方向连接墙面则拒绝；
- 禁止跨越凹口的任意角度斜撑；
- 禁止普通轴压支撑终止于另一支撑跨中；
- 无法闭合时仅返回一个 `controlled_block` 诊断方案，不再返回三张近似失败卡片；
- 前端明确显示平面分类和环撑、中心岛、分区施工或显式双向框架建议。

受控阻断属于安全结果，不具备计算和施工版发行资格。

## 6. 服务器离线根因

服务器监控显示 CPU 100%、系统负载约 103、内存使用约 85.6%。这表明主要故障不只是物理内存不足，还包括：

- 完整计算与 HTTP 服务处于同一个 Uvicorn 进程；
- 支撑候选搜索在 API 进程内同步执行；
- 重复点击可提交多个等价重任务；
- A/B/C 候选可并行建立多套项目深拷贝和密集矩阵；
- NumPy/BLAS、Python线程和基准算例子进程可能造成 CPU 过度订阅；
- 计算异常、原生库崩溃或 OOM 会直接带走 API；
- 固定 6.5 GB 服务内存上限不会随服务器扩容变化。

增大内存只能推迟部分 OOM，无法解决同进程计算导致的 CPU 饥饿和故障域耦合。

## 7. 隔离运行架构

生产环境新增 `pitguard-worker.service`：

```text
Nginx
  └─ pitguard-api.service
       ├─ 登录、项目、任务和静态接口
       └─ SQLite任务队列
             └─ pitguard-worker.service
                  └─ 每次只运行一个工程重任务
```

关键控制：

- API 使用 `PITGUARD_TASK_EXECUTION_MODE=external`，不创建计算线程池；
- worker 使用 `PITGUARD_TASK_EXECUTION_MODE=worker`；
- 支撑候选优化、完整计算、A/B/C、IFC、钢筋和完整交付均由 worker 执行；
- 任务按项目、操作和载荷哈希去重，重复点击返回同一任务；
- SQLite 使用原子领取，避免两个进程同时计算同一任务；
- worker 默认一次只执行一个重任务；
- 每个任务完成后 worker 主动退出，由 systemd 启动全新进程，彻底释放 NumPy、Matplotlib和原生分配器内存；
- 默认硬超时 1800 s，超时后任务标记为中断并退出 worker；
- worker CPU 默认限制为 600%，数值库线程默认 1；
- worker 内存上限按服务器内存 60%自适应，最低 3 GB、最高 32 GB；
- API 默认 `MemoryHigh=2G`、`MemoryMax=4G`，并设置较低 OOM 分数，优先保持网页在线；
- worker 设置较低 CPU/IO 权重和较高 OOM 分数，资源压力下优先终止计算进程；
- 同步重计算接口在生产模式下改为排队或返回明确的任务提交提示。

## 8. 部署要求

V3.27.0 首次升级必须运行：

```bash
cd /opt/excavation-design
sudo bash start-linux.sh
```

只执行 `restart-production.sh` 无法创建新的 worker unit 和资源控制参数。

部署后检查：

```bash
sudo systemctl status pitguard-api pitguard-worker --no-pager
sudo journalctl -u pitguard-api -n 100 --no-pager
sudo journalctl -u pitguard-worker -n 200 --no-pager
systemd-cgtop
curl http://127.0.0.1:8002/health
```
