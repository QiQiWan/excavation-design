# PitGuard V3.59.0：最小设计任务书与渐进式资料门禁

## 结论

本版本不减少计算、验算或配筋能力，而是重新划分资料的使用时点。设计入口由二十余项专业参数改为四项工程师决策，系统内部仍保留完整参数、来源、专业覆盖和人工改写能力。

## 新的默认流程

1. **快速方案**：确认成果目标、周边保护要求、设计取向和结构使用阶段；录入闭合轮廓与最终开挖深度；生成 A/B/C 围护方案。
2. **计算与优化**：方案确定后只补地勘、水位等高影响资料；系统建立地质设计域、按支撑标高自动生成施工阶段，并执行计算—校核—补强—复算。
3. **配筋与交付**：基于当前有效施工阶段内力包络完成墙、围檩、水平支撑和节点配筋；正式出图前再关闭承载力、专项设计、校审与发行门禁。

“专业流程”仍显示设计基准、轮廓与深度、快速方案、计算验算、配筋深化和成果交付六个技术步骤。

## 资料分层

| 层级 | 典型资料 | 是否阻断快速方案 | 使用时点 |
|---|---|---:|---|
| 现在必须 | 四项设计意图、闭合轮廓、最终开挖深度 | 是 | 方案生成 |
| 系统推荐 | 规范组合、首轮材料、分析模型、施工阶段 | 否 | 系统带来源生成，工程师可改 |
| 计算前补 | 钻孔/地层、地下水位、计算用地质设计域 | 否 | 内力、位移、稳定计算前 |
| 相关验算前补 | 立柱基础承载力、换撑刚度等 | 否 | 采用对应构造时 |
| 正式出图前补 | 专项设计、校审、审批、正式不利工况证据 | 否 | 正式发行 |

## 后端变更

- 新增 `design_intake` 服务，统一输出资料层级、可执行动作、三阶段状态和下一步原因。
- 新增 `/api/projects/{project_id}/design/intake/apply`，把四项工程师选择转换为可审计设计设置。
- 候选方案生成的输入门禁改为“已确认任务书 + 闭合轮廓与深度”；地质资料继续严格控制正式计算。
- 不再把未录入的立柱基础承载力误列为全部设计基准的前置阻断。

## 前端变更

- 默认显示“快速方案—计算与优化—配筋与交付”三阶段，六个技术步骤移入专业流程。
- 设计基准页改为四问式最小设计任务书；完整专业参数按需展开。
- 工程输入按“现在必须”和“正式计算前再补”分组。
- 有钻孔但尚无地质设计域时，计算按钮自动先建模再计算。
- 计算入口明确显示四项准备状态，并以中文说明系统自动完成的动作。

## 工程边界

- 推荐值不是项目实测值；地下水、地勘和承载力不会被系统虚构。
- 快速方案只用于方案比较，不能绕过正式计算与发行门禁。
- 所有自动补强仍受既有单调增强、可追溯和最大迭代次数约束。

## 验证结果

- V3.59 渐进式输入与 V3.55 智能闭环定向后端回归：11 项通过。
- 前端组件与流程回归：21 个测试文件、39 项测试全部通过。
- 前端生产构建通过。
- 全量后端历史套件：371 项通过、3 项跳过、24 项失败。抽取其中地质外延、候选方案数量、凹形轮廓支撑拓扑等 4 项回放到原 V3.55，均同样失败，确认并非本次流程改造引入；这些既有计算内核债务仍需后续专项修复，不能据此宣称全量历史回归已经清零。
