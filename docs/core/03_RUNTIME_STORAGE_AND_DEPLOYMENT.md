# 运行时、存储与部署

## 前端职责

前端只负责输入、候选预览、当前结果摘要、任务进度和成果下载。核心页面不会自动请求工业成熟度、监测、完整问题中心、基准算例或多套深化数据。

## API 职责

API 返回项目摘要、工作区投影、核心状态、候选预览和任务状态。普通页面查询不重组完整工程对象，不执行完整计算。

## Worker 职责

后台任务负责候选生成、施工阶段计算、配筋、IFC、CAD 和计算书。重任务根据实时内存和 CPU 状态串行或有限并发执行。

## 存储分层

- 项目主快照：当前设计状态和对象引用；
- 工作区投影：网页所需的有界数据；
- 外部对象：完整地质面、阶段结果、候选完整计算、逐根钢筋和导出数据；
- 任务表：进度、日志和结果文件路径；
- 修订表：审计所需的不可变版本。

## 数据控制

默认保留一个当前完整计算结果，历史结果只保留摘要。候选最多三个，完整方案计算默认一个。外部对象采用内容寻址，相同内容不会重复保存。

## 候选搜索内存边界

Core 模式候选搜索默认最多试算 9 组，内存中最多保留 6 套完整候选，最终保存不超过 3 套。试算模型从干净围护墙种子构建，不复制历史候选、计算结果、配筋和审计数据。方案采用属于独立 worker 任务，不会在 API 中重新执行候选搜索。

Windows 默认对一次性 worker 安装 6 GB Job Object 进程内存硬上限，并每秒采集 RSS、Private Bytes、峰值工作集和系统可用内存。可通过 `PITGUARD_WORKER_DEFAULT_HARD_CAP_MB` 调整行政上限，桌面工作站不建议超过物理内存的 25%。

运行诊断写入 `runtime/diagnostics`，可执行：

```bash
python scripts/summarize-runtime-diagnostics.py --runtime runtime
```

查看任务阶段、候选试算、项目序列化和 worker 峰值内存。
