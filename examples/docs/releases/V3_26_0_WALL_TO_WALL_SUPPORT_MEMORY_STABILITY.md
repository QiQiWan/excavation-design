# PitGuard V3.26.0 端部墙—墙传力与内存稳定性

## 1. 问题判定

平面图中沿竖向绘制的蓝色构件仍属于水平支撑，只是其平面轴线方向为 Y 向。现有全局计算将普通水平支撑建模为以轴向受压为主的墙—墙杆件。绿色端墙短撑若终止于蓝色主支撑跨中，会向主支撑施加平面内横向集中力。临时立柱主要提供竖向承托和支撑平面外稳定，不能自动提供该节点所需的平面内水平反力，也不能使蓝色主支撑自动具备传递该横向荷载的梁式弯剪刚度。

因此，除非建立显式双向框架模型，并完成节点刚度、主支撑平面内弯剪、节点板/钢筋、立柱与立柱桩水平反力等设计，普通支撑不得终止于另一根支撑中部。

## 2. V3.26.0 支撑拓扑规则

普通非环形支撑必须满足：

- 两端分别落在围护墙、围檩或环梁的有效节点；
- 同层非法跨中穿越为零；
- `supportToSupportTerminalCount = 0`；
- `unsupportedInternalEndpointCount = 0`；
- 角撑族不得形成 V 形或扇形；
- 多根角撑不得共用或挤占同一墙上承压节点。

端墙支点不足时，修复顺序为：

1. 保留可行的直接墙—墙对撑；
2. 从端墙控制站位向周边墙面搜索两端落墙的长斜撑；
3. 在共同转角两侧生成独立、近似平行的角撑族；
4. 若仍无法满足围檩支点间距，则安全阻断，要求采用环撑、中心岛或显式双向框架。

不再自动生成“端墙短撑—主支撑跨中 T/Y 节点”。旧项目中的该类构件在围檩修复时会被识别并移除。

## 3. 交点最少目标

候选方案采用词典序主目标，优先级为：

1. 支撑中部终止点数量；
2. 无支承内部端点数量；
3. 非法同层穿越数量；
4. 角撑扇形异常和墙节点拥挤数量；
5. 墙上高分支汇交数量；
6. 墙上汇交、内部汇交和总平面复杂度；
7. 辅助杆件及支撑总数；
8. 轴力、跨度、对称性、立柱数量和施工通道。

这堵住了旧逻辑中“将交叉杆截短至主支撑，使非法交叉变成合法 T/Y 汇交”的指标漏洞。

## 4. 计算门禁

计算开始前重新评价支撑传力路径。出现以下问题时直接拒绝进入求解器：

- 支撑终止于另一普通支撑；
- 内部端点没有墙、围檩、环梁或受支持的显式结构节点；
- 非环形同层非法穿越；
- 构件中心线离开基坑范围。

报错会明确要求改用直接对撑或墙—墙长斜撑。系统不会为无效节点分配虚构的轴向支座反力。

## 5. 通用平面资格边界

旋转矩形和梯形可形成直接墙—墙支撑时，资格结果为 `calculation_ready`。

对近方形、大开口 U 形或深凹平面，如果当前轴压墙—墙拓扑无法同时满足围檩支点间距、零穿越和有效端点要求，结果为 `controlled_block`：

- 资格测试判定系统行为安全；
- 项目本身 `calculationReady=false`；
- 不允许计算和施工版发行；
- 推荐环撑、中心岛或显式双向框架。

该口径避免为了让测试全绿重新引入不成立的 T/Y 伪支座。

## 6. 内存稳定性改造

服务器崩溃的主要风险来自完整计算、候选方案深拷贝、密集 NumPy 矩阵、多个候选并行、历史完整结果序列化和重型三维组件同时挂载。

V3.26.0 默认参数：

```text
PITGUARD_TASK_WORKERS=2
PITGUARD_HEAVY_TASK_CONCURRENCY=1
PITGUARD_CANDIDATE_WORKERS=1
PITGUARD_CALCULATION_RESULT_RETENTION=1
PITGUARD_TASK_MEMORY_SOFT_LIMIT_MB=5600
MALLOC_ARENA_MAX=2
MemoryHigh=5G
MemoryMax=6500M
```

具体措施：

- 一键计算只计算当前采用方案，`topN=0`；
- A/B/C 完整计算由独立按钮触发；
- 重任务通过全局信号量串行；
- 同一项目写任务通过项目锁串行；
- 候选方案默认串行计算，每个候选结束后执行 GC；
- 重任务启动前检查 RSS，超过软上限时受控终止并保留 API 服务；
- 任务结束后执行 Python GC 和 Linux `malloc_trim(0)`；
- 完整计算结果只保留最近一次，历史结果转为轻量审计摘要；
- 计算结果中的三维云图和高级结果组件仅在用户展开后挂载；
- systemd 通过 `MemoryHigh` 和 `MemoryMax` 保护宿主机。

## 7. 部署

本版本修改了后端、前端生产构建和 systemd 环境。覆盖代码后首次必须运行：

```bash
cd /opt/excavation-design
sudo bash start-linux.sh
```

仅运行 `restart-production.sh` 不会更新前端构建和 systemd 内存参数。

检查：

```bash
sudo systemctl status pitguard-api --no-pager
curl http://127.0.0.1:8002/health
curl -s https://designer.eatrice.cn/api/system/metrics
sudo journalctl -u pitguard-api -f
```

指标接口会返回当前 RSS、软内存上限、任务工作线程和重任务并发数。
