# PitGuard V3.29.0 布设方案设计器完整性与运行韧性优化报告

## 1. 评估结论

V3.28.0 已经建立了“轮廓识别—体系选型—候选生成—拓扑硬约束—完整计算—受控交付”的基本链路，且对狭长矩形、旋转矩形、紧凑凸平面、圆/椭圆竖井及 L/U/T/C/Z/H 等凹形平面具有差异化处理。其主要问题已经从“是否具有功能”转移到以下三个工业化层面：

1. **设计决策链缺少统一审计合同**：形状识别、体系兼容、墙长变量、候选多样性、施工阶段、计算资源和正式发行分别存在，但用户无法直接确认整个链路是否闭合。
2. **复杂项目的搜索和计算资源上界不明确**：异形轮廓候选组合、项目深拷贝、完整结果序列化和重型导出会叠加内存与 CPU 压力。
3. **API 短时不可用时前端恢复能力不足**：截图中静态前端已经加载，DOM 停留在 `main.loginLoading`，浏览器无明显脚本错误，说明前端正在等待登录状态接口；计算进程引发 API 重启或无响应后，页面长期停留在“正在验证登录状态”。

V3.29.0 将设计器完整性和运行韧性作为同一闭环处理。当前系统可以定位为：

- 规则型和启发式的高级方案设计辅助系统；
- 对支持范围内的平面能够自动形成受力可解释的候选；
- 对超出当前轴压杆/环撑模型能力的平面执行受控阻断；
- 重计算失败、超时、OOM 或原生库异常时，API、登录和项目管理保持独立运行；
- 仍未达到“任意异形平面全自动施工图签发”的法定工业系统水平。

## 2. 当前布设方案设计器的合理性

### 2.1 已经合理落实的能力

1. **形状识别先于支撑生成**
   - 使用最小旋转外接矩形建立局部长轴和短轴；
   - 计算长宽比、圆度、矩形度、凸度、正交边比例和凹角数量；
   - 识别矩形、近方形、梯形、平行四边形、圆形、椭圆形、凸多边形及 L/U/T/C/Z/H 等凹形轮廓；
   - V3.29.0 新增识别置信度和歧义原型提示。

2. **体系族和形状原型关联**
   - 狭长平面优先短跨直对撑；
   - 短墙端部采用独立落墙的平行角撑族；
   - 近方形、圆形和椭圆形采用闭合内环与径向撑；
   - 正交凹形采用走廊分区支撑；
   - 凹角或多臂交汇区没有转接环、分隔墙、中心岛或显式框架时保持受控阻断。

3. **传力路径硬约束**
   - 普通轴压支撑两端必须落在围护墙、围檩或环梁；
   - `supportToSupportTerminalCount` 必须为 0；
   - `unsupportedInternalEndpointCount` 必须为 0；
   - 同层非法穿越必须为 0；
   - 支撑中心线必须位于坑内；
   - 围檩无支承跨必须满足项目硬限值。

4. **洁净支撑目标已经进入完整链路**
   - 非法穿越；
   - 墙上高分支汇交；
   - 墙上汇交；
   - 内部 T/Y/X 汇交；
   - 平面交叉复杂度；
   - 辅助杆件数量和总杆件数量；
   - 以上指标进入生成、硬约束、词典序排序、前端显示和正式发行门禁。

5. **围护墙设计变量已经进入方案台账**
   - 计算控制段长度；
   - 施工槽段长度和数量；
   - 局部加强范围；
   - 墙趾标高和分区；
   - 支撑拓扑和墙长变量可进行联合评价。

6. **候选方案能够执行完整计算**
   - 候选快速筛选和完整计算分层；
   - 当前采用方案默认单独计算；
   - A/B/C 完整计算按需逐个提交；
   - 结果绑定输入、采用设计和算法版本哈希。

### 2.2 尚不完整的能力

1. **体系识别仍以规则阈值为主**
   - 临界长宽比、近方形和不规则四边形可能位于多个分类边界；
   - 当前已经给出置信度和备选原型，但尚未对所有备选原型自动执行多模型比选。

2. **凹形平面的转接结构尚未全自动设计**
   - 可以识别 L/U/T/C/Z/H 形并生成分区支撑；
   - 转接环梁、分隔墙、中心岛和显式平面框架的尺寸设计、节点设计及整体计算尚未全部自动化。

3. **候选生成仍属于有界启发式搜索**
   - 当前方法具有工程可解释性和硬约束控制；
   - 尚未形成严格意义上的混合整数拓扑优化、图搜索全局最优证明或可靠度优化。

4. **施工性输入仍依赖项目录入质量**
   - 未录入坡道、出土口、地下室柱网和保护区时，施工性评分只能低置信度运行；
   - 临时立柱、吊装、安装顺序和出土通道仍需进一步联合优化。

5. **独立计算复核仍在同一代码体系内**
   - 已有双路径对账和计算合同；
   - 法定成果仍需独立程序、商业软件或人工复核。

## 3. V3.29.0 完整性审计

新增 `support_scheme_designer_audit.py`，将设计器拆成十个可审计分项：

1. 输入与设计域；
2. 形状识别；
3. 体系选型与转接；
4. 传力与拓扑；
5. 围护墙设计变量；
6. 候选多样性；
7. 施工与节点可实施性；
8. 施工阶段一致性；
9. 计算合同与发行；
10. 计算资源预算。

审计结果提供：

- 总状态：`pass / warning / blocked`；
- 0～100 分完整性评分；
- 阻断项和警告项；
- 形状原型与识别置信度；
- 候选数量、拓扑族数量、实质几何方案数和硬可行方案数；
- 围护墙设计长度覆盖率、施工槽段数量和复算状态；
- 计算资源预算；
- 当前模型兼容性和完整工作流。

前端新增分项状态条，设计人员可以直接判断设计器缺口位于几何、体系、拓扑、墙长、施工、计算或发行阶段。

## 4. 候选搜索和资源上界

V3.29.0 增加两类硬上限：

- `PITGUARD_SUPPORT_CANDIDATE_TRIAL_LIMIT`，默认 36；
- `PITGUARD_MAX_SUPPORT_ELEMENTS`，默认 2400。

候选生成器在进入深拷贝、质量评价和序列化前检查杆件规模，避免复杂凹形平面产生组合爆炸。

新增保守资源估算器，计算：

- 施工阶段数量；
- 围护墙段数量；
- 支撑、围檩和立柱数量；
- 地质表面网格单元数；
- 单墙面自由度估计；
- 候选完整计算复制倍率；
- 估算峰值内存与 worker 上限比值。

资源状态分为：

- `normal`：允许正常计算；
- `elevated`：建议逐方案计算；
- `high`：强制安全模式；
- `blocked`：计算前受控阻断。

## 5. 计算后整站离线的根因与修复

### 5.1 根因

1. API 和重计算过去位于同一进程或同一资源域；
2. 候选搜索、矩阵、项目深拷贝和结果序列化同时占用内存；
3. Python、NumPy、BLAS 和 OpenMP 线程叠加；
4. 任务结束后原生内存分配器不一定立即归还内存；
5. SQLite 项目正文和修订快照反复写入多 MB 结果；
6. 任务列表返回完整结果对象；
7. 前端登录状态和任务轮询缺少快速失败与恢复路径。

### 5.2 进程隔离

生产架构现在为：

```text
Nginx
  ├── 静态前端
  └── pitguard-api.service
        ├── 登录/权限
        ├── 项目管理
        ├── 任务队列
        └── 轻量查询

pitguard-worker.service
  └── 候选搜索/完整计算/IFC/图纸/钢筋/交付
```

worker 每完成一个重任务即退出，由 systemd 启动新进程，从操作系统层面清理 NumPy、Matplotlib、BLAS 和第三方原生库内存。

### 5.3 资源看门狗

worker 运行期间持续监控：

- 当前进程 RSS；
- 系统 `MemAvailable`；
- 任务心跳；
- 硬超时。

触发以下任一条件时，当前任务标记为 `interrupted`，worker 退出，API 保持在线：

- RSS 超过 `PITGUARD_WORKER_RSS_HARD_LIMIT_MB`；
- 系统可用内存连续低于保留值；
- 任务超过 `PITGUARD_TASK_TIMEOUT_SECONDS`；
- systemd cgroup 达到 `MemoryMax`；
- 原生库崩溃或 worker 被 OOM 杀死。

### 5.4 动态 cgroup 预算

一键部署脚本基于服务器实际 `MemAvailable`，而非只看总内存，计算 worker 上限：

```text
worker预算 = min(总内存50%, 当前可用内存-系统保留值, 32GB)
```

默认还设置：

- `MemorySwapMax=0`；
- `CPUQuota=300%`；
- `Nice=5`；
- `IOWeight=20`；
- `IOSchedulingClass=idle`；
- `OOMScoreAdjust=800`。

API 使用独立低内存预算和 `OOMScoreAdjust=-500`，资源紧张时优先保留登录和项目服务。

## 6. API 启动与数据库减负

1. 计算、IFC、CAD、DOCX、钢筋和基准算例模块改为按请求/任务延迟导入；
2. 当前环境下 `import app.main` 峰值 RSS 约 333 MB；
3. 项目列表读取独立 `summary` 列，不解析完整项目 JSON；
4. SQLite 使用 WAL、`temp_store=FILE`、受控缓存和无 mmap；
5. 完整计算任务只在结束时保存一次项目修订；
6. 默认只保留最近一次完整计算结果，历史结果保存轻量审计摘要；
7. 任务列表默认不返回大体量 `result`；
8. worker 心跳写入 `runtime/worker-heartbeat.json`。

## 7. 前端不再卡在登录验证

截图中的页面停留在：

```text
正在验证登录状态…
```

V3.29.0 调整为：

1. `/api/auth/status` 超时 5 秒；
2. Nginx 对该接口使用 2～3 秒连接/读取超时；
3. 接口不可达时进入应用登录页，显示“登录服务暂不可用”和重试按钮；
4. 每 15 秒自动重新检查服务；
5. 页面不再无限显示加载状态；
6. 新增 React 错误边界，渲染异常时可重载或清理瞬态状态；
7. API 返回 401 时自动跳转登录页；
8. `/health/live` 仅检查 API 进程；
9. `/health/ready` 以 1 秒数据库超时检查就绪状态。

## 8. 任务断线恢复

计算和导出任务现在具有：

- 重复提交哈希去重；
- `queued/running/success/failed/cancelled/interrupted` 完整状态；
- 任务 ID 保存至 `sessionStorage`；
- 浏览器刷新后自动恢复后台任务轮询；
- API 短时不可用时允许 8 次重连；
- 页面停止轮询时不会重复提交任务；
- worker 超时或 OOM 后，前端进入明确失败状态；
- 导出卡片同样把 `interrupted` 识别为终态；
- 完成后项目刷新失败不会否定已完成的计算任务。

## 9. 优化后的工业级设计工作流

```text
1. 输入冻结与设计域检查
2. 轮廓清理、局部主轴和形状识别
3. 识别置信度与歧义原型确认
4. 支撑体系和异形转接体系选型
5. 有界候选生成
6. 墙—墙传力、坑内包含、交叉和围檩跨硬约束
7. 围护墙设计长度、施工分幅和墙趾联合变量
8. 坡道、出土口、柱网、保护区和人工锁定检查
9. 施工阶段激活、换撑和拆撑一致性检查
10. 计算资源预检
11. 独立 worker 完整计算
12. 独立路径对账和规范追溯
13. 方案采用、节点和配筋深化
14. IFC/CAD/计算书/钢筋包一致性检查
15. 审签和受控发行
```

## 10. 仍需继续优化的工业能力

### P0：真实工程验证

- 建立矩形、梯形、L/U/T/C/Z/H、曲线墙、多坑和带内岛项目基准库；
- 与商业软件和真实设计成果对标；
- 建立误差限值、审查记录和回归数据集。

### P1：异形转接结构自动化

- 自动设计局部环梁、多环和转接桁架；
- 自动生成分隔墙和中心岛；
- 支撑节点平面内弯剪和刚域模型；
- 多分区施工时序联合优化。

### P2：真正的全局优化

- 采用图搜索、混合整数规划或分层拓扑优化；
- 将结构安全、变形、交点、节点复杂度、材料量、施工通道和工期纳入统一多目标；
- 引入参数不确定性、敏感性和可靠度分析。

### P3：高可用部署

单机进程隔离能够防止计算任务拖死 API，但无法抵御整机、磁盘、内核、网络或电源故障。更高可用级别需要：

- 两个及以上 API 节点；
- 负载均衡和健康摘除；
- PostgreSQL；
- Redis/RabbitMQ/Celery 或同等级任务队列；
- 对象存储；
- worker 池和任务租约；
- 数据库主从、备份和灾备恢复演练；
- Prometheus/Grafana/告警系统。

因此，“任何情况下不离线”无法由单机应用代码绝对保证。V3.29.0 实现的是：在配置资源边界内，计算、候选搜索和导出失败时受控降级，API和静态前端尽可能保持在线；整机级故障需要高可用基础设施。

## 12. 部署要求

V3.29.0 修改了前端、API、worker、systemd 和 Nginx，首次升级必须执行：

```bash
cd /opt/excavation-design
sudo bash start-linux.sh
```

仅执行 `restart-production.sh` 不会更新 systemd 资源参数、Nginx 快速健康路由和生产前端。

运行检查：

```bash
bash status-production.sh
sudo journalctl -u pitguard-api -f
sudo journalctl -u pitguard-worker -f
systemd-cgtop
```
