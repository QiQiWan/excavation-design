# PitGuard V3.24.0：工业级计算基线与受控交付

## 1. 版本目标

V3.24.0 将“计算完成”拆分为可计算、计算可信、可审查、可发行和可归档五个独立状态。系统不再以是否存在计算结果作为施工版发行依据，而以不可变计算合同、数值质量、独立复核、规范追溯、岗位审签和成果内容根共同决定发行资格。

## 2. 工业计算状态机

| 状态 | 核心条件 | 失败后的行为 |
|---|---|---|
| D0 可编辑草案 | 项目已创建 | 允许编辑，不允许正式发行 |
| D1 输入可计算 | 单位、几何、地质、材料、工况与构件引用有效 | 阻断求解或降级为明确标识的筛查 |
| D2 输入基线冻结 | 形成求解前输入哈希、工况哈希、几何哈希、拓扑哈希 | 任一计算相关输入变化后旧结果失效 |
| D3 分阶段求解完成 | 所有施工阶段、墙段和激活构件均有结果 | 缺阶段、缺墙段、NaN/Inf 或求解失败时阻断 |
| D4 计算质量通过 | 条件数、残差、回退求解、覆盖率和独立复核满足阈值 | 形成 fail/warning，不进入正式审签 |
| D5 规范与构造校核通过 | 条文来源、对象、数值、限值和状态可追溯 | 关键条文缺失或失败时阻断 |
| D6 审签批准 | 设计、校核、审核、批准岗位链完整且绑定当前修订 | 修改项目或重算后批准失效 |
| D7 受控发行 | 计算合同、结果哈希、当前修订和全部成果内容根一致 | 施工版在生成前阻断，审查版可带降级标识输出 |
| D8 归档可验证 | 发行证书、文件哈希、内容根和审计记录完整 | 文件被替换或遗漏时校验失败 |

## 3. 计算合同

每次完整计算同时保存两类快照：

1. 求解前输入快照：设计依据、单位、坐标系、地质、开挖、围护体系、施工工况、荷载和计算控制参数；
2. 采用设计快照：自动截面迭代、墙趾优化和配筋设计结束后实际采用的构件参数。

计算合同包含：

- `inputSnapshotHash`；
- `caseHash`；
- `geometryHash`；
- `supportTopologyHash`；
- `adoptedDesignSnapshotHash`；
- `calculationContractId`；
- `resultHash`；
- 软件、算法、规则集和导出模式版本；
- Python、NumPy、操作系统、CPU架构、字节序和数值线程配置。

钢筋几何导出数量、三维显示密度、监测预警阈值、工作区模式和发行审批策略属于交付或运行配置，不进入结构计算输入哈希。荷载、材料、几何、施工阶段、地下水、支撑、墙体和计算阈值变化仍会使计算合同失效。

## 4. 输入与施工阶段审计

统一输入审计检查：

- 单位与坐标系统；
- 开挖边界闭合和有效性；
- 地质覆盖范围和深度；
- 墙、围檩、支撑、立柱的几何与材料参数；
- 支撑端点、拓扑和构件引用；
- 施工阶段编号唯一性；
- 开挖标高单调性；
- 构件激活、停用和再次激活逻辑；
- 支撑安装标高与支撑层标高一致性；
- 地下水、附加荷载和阶段参数有限性。

阶段工况存在反向开挖、无效支撑引用、激活与停用重叠、提前激活或非法再激活时，输入质量状态为 fail。

## 5. 数值质量与独立复核

计算质量包统一汇总：

- 阶段×墙段结果覆盖率；
- 矩阵条件数；
- 相对平衡残差；
- 回退求解次数；
- NaN/Inf检查；
- 全局耦合墙体位移与参考墙体分析的差异；
- 支撑轴力与独立对账路径的差异；
- 规范校核记录的追溯完整率。

阈值可在项目高级设置中配置：

- 最大矩阵条件数；
- 最大相对平衡残差；
- 独立复核预警差异比；
- 独立复核失败差异比。

当前独立复核属于同一系统内部的双路径对账，可识别大部分模型装配、符号、激活和结果映射错误。法定工程签发前仍应引入独立代码库或第三方软件对标。

## 6. 受控交付

施工版协同交付在文件生成前检查：

- 当前项目修订；
- 当前计算合同；
- 输入哈希和采用设计哈希；
- 计算质量状态；
- 正式化闸门；
- 设计、校核、审核、批准状态；
- 施工版批准修订是否与当前修订一致。

发行包增加：

- `calculation_contract.json`；
- `calculation_assurance.json`；
- `delivery_readiness.json`；
- `release_certificate.json`；
- 成果文件哈希清单；
- 全部成果内容根哈希；
- 交付关系矩阵和移交清单。

IFC属性集、IFC侧车清单、DOCX计算书、施工图追溯JSON和钢筋深化包均写入同一计算合同ID、输入哈希、采用设计哈希和结果哈希。

## 7. 钢筋深化交付性能

旧流程会在多个JSON中重复保存逐根钢筋数组，并使用常规XLSX对象模型和高压缩级别。约6万根钢筋的导出容易超过5分钟。

V3.24.0采用：

- XLSX写入只读流式模式；
- 紧凑语义JSON；
- 单独的完整逐根几何JSON；
- CSV作为完整机器表；
- 快速ZIP压缩级别；
- 大数组只保存一次，并在主清单中记录外部文件引用。

基准算例约6万根钢筋的深化包导出为12.509秒，压缩包约8.11 MB。性能会随服务器CPU、磁盘和钢筋复杂度变化。

## 8. 接口和前端

新增或强化接口：

- `GET /api/projects/{project_id}/calculation/assurance`；
- `GET /api/projects/{project_id}/export/release-readiness`；
- 施工版导出在发行基线失败时返回HTTP 409。

前端计算恢复面板显示：

- 计算质量总状态；
- 计算合同ID；
- 阶段覆盖率；
- 条件数、残差和回退次数；
- 独立复核差异；
- 规范追溯率；
- 输入、采用设计和结果哈希；
- 阻断项、警告项和整改动作。

## 10. 工业应用边界

V3.24.0完成了软件内部的计算基线、质量门禁和受控交付闭环。以下工作仍是设计单位正式生产应用前的必要条件：

1. 建立真实工程和第三方软件基准库；
2. 对软土、承压水、复杂换撑、邻近建构筑物和异形基坑开展专项验证；
3. 引入独立求解器或外部复核程序；
4. 建立规范条文版本库及法务审查；
5. 完成生产数据库、对象存储、任务队列、容灾与高可用部署；
6. 采用成熟IFC SDK、IDS和BCF验证；
7. 建立数字签章、时间戳、WORM归档和撤销机制；
8. 由注册岩土/结构工程师和设计单位质量体系完成审签。
